## 슬라이드별 대본

### `T00`
- 20초:
  - "오늘은 4주차 오전과 5~6주차 오후를 한 흐름으로 연결해서, 실제로 돌아가는 실습 구조를 보여드리겠습니다."
- 60초:
  - "강사님 이론 뒤에 바로 붙어서 보조강사 파트는 실습 중심으로 진행합니다. 오전에는 환경과 개발 프로세스 기준선을 맞추고, 오후에는 데이터 분석/코드/웹 시연으로 이어집니다."

### `M01`
- 20초:
  - "오전은 워밍업, 질문 정리, 환경 Q&A, SDLC 순으로 갑니다."
- 60초:
  - "통화에서 합의한 대로 먼저 긴장 완화와 질문 해소를 하고, 개발환경 편차를 정리한 다음 4주차 소프트웨어 개발 프로세스를 실무자 관점으로 설명하겠습니다."

### `M02`
- 20초:
  - "개발환경은 로컬 기준으로 표준 경로를 먼저 맞추겠습니다."
- 60초:
  - "오늘은 로컬 환경에서 바로 실습 가능한 상태를 만드는 데 집중하겠습니다. 표준 경로는 Cursor 권장, VS Code+Claude Code 허용이며, 수업 중 즉시 대응은 이 두 조합으로 한정합니다. 핵심은 IDE를 완전히 같게 하는 것이 아니라 로컬 실행 절차를 동일하게 맞추는 것입니다. 원격 처리는 이후 각자 환경에서 진행하겠습니다."

### `M03`
- 20초:
  - "실습 안정성을 위해 IDE는 표준 1개로 먼저 통일하겠습니다."
- 60초:
  - "오늘 실습 기본 IDE는 Cursor로 통일하고, 설치 이슈가 있는 경우에만 VS Code fallback으로 진행하겠습니다. 중요한 건 도구 취향이 아니라 동일한 실행 절차를 맞추는 것입니다. 노트북 성능 편차는 허용하되 체크리스트는 하나로 맞추고, 질문은 즉시 받아 우회 경로를 안내하겠습니다."

### `M04`
- 20초:
  - "AI 코딩 결과를 통제하기 위한 규칙 체계를 먼저 보여드립니다."
- 60초:
  - ".cursorrules, docs/rules, 세션 문서가 각각 역할을 나눠서 동작합니다. 핵심은 문서(계약), Gate(상태), 테스트(증적)를 결합해 AI 생산성을 품질과 함께 가져가는 것입니다."

### `M05`
- 20초:
  - "Gate A~E는 승인 범위를 분리하는 상태 머신입니다."
- 60초:
  - "A에서 계획 승인, B에서 구현 완료 후 멈춤, C에서 테스트 계획 승인 후 실행으로 권한을 나눕니다. 이 구조를 지키면 승인 과대해석과 품질 사고를 크게 줄일 수 있습니다."

### `M06`
- 20초:
  - "4주차 커리큘럼 항목이 실제 프로젝트에서 어떻게 구현되는지 연결하겠습니다."
- 60초:
  - "AI 도구로 코드 작성/리뷰/피드백을 수행하고, 리팩토링과 테스트 자동화를 함께 운영한 사례를 보여드립니다. 이론 항목이 실습에서 어떻게 작동하는지 확인하는 시간입니다."

### `M07`
- 20초:
  - "오전 마지막은 API 계약 확인입니다."
- 60초:
  - "공인 URL의 Swagger와 OpenAPI를 열어 SCM/CRM/BI/Agent 구조를 확인합니다. 이 단계가 끝나면 오후 실습에서 화면 결과를 API와 바로 연결해서 설명할 수 있습니다."

### `M08`
- 20초:
  - "오전 합의점을 고정하고 오후 데이터 실습으로 전환합니다."
- 60초:
  - "오전에는 환경/규칙/프로세스를 맞췄고, 이제 오후에는 실제 데이터와 코드 흐름으로 들어갑니다. 질문은 구간별로 정리해 흐름이 끊기지 않게 운영하겠습니다."

### `A01`
- 20초:
  - "오후는 5~6주차 목표를 통합해 실습 중심으로 진행합니다."
- 60초:
  - "5주차의 데이터 과학 AI 도구 활용과 6주차 프로젝트 실습을 한 번에 연결합니다. 화면 결과를 보여주고 코드와 알고리즘 설명으로 이어지는 구조입니다."

### `A02`
- 20초:
  - "데모 동선을 먼저 고정합니다."
- 60초:
  - "접속, CSV 업로드, dataset_id 확인, 탭 실행, Dify 요약 순서로 진행합니다. 기능 설명보다 동선 안정성을 우선해 시연 성공률을 높입니다."

### `A03`
- 20초:
  - "2-Tier 분리를 다시 확인합니다."
- 60초:
  - "Tier1은 계산 엔진, Tier2는 해석 엔진입니다. 역할 분리가 명확할수록 운영 안정성과 확장성이 좋아집니다."

### `A04`
- 20초:
  - "SCM은 수요예측 차트와 의사결정 연결이 핵심입니다."
- 60초:
  - "SCM 샘플 CSV를 기반으로 예측 라인을 확인하고, 재고/수요 타이밍 의사결정으로 연결합니다. 알고리즘은 Prophet 중심, 필요 시 ARIMA fallback 흐름으로 설명합니다."

### `A05`
- 20초:
  - "CRM은 이탈 위험 고객을 우선순위로 보여줍니다."
- 60초:
  - "Top-N 위험 고객과 요약 정보를 통해 영업 액션 우선순위를 빠르게 잡을 수 있습니다. RFM과 K-Means가 어떤 역할을 하는지 코드 설명 구간에서 함께 다룹니다."

### `A06`
- 20초:
  - "BI는 지역/시기 트렌드 인사이트를 제공합니다."
- 60초:
  - "히트맵과 YoY 비교로 시기별 수요 변화를 확인하고, 지역 전략 기획으로 연결합니다. GroupBy/Pivot 기반 집계를 실무 인사이트로 해석하는 예시입니다."

### `A07`
- 20초:
  - "이제 화면 결과를 만든 코드 흐름으로 넘어갑니다."
- 60초:
  - "Endpoint부터 Service, compact 응답, Dify 연계까지 핵심 라인만 짚습니다. 전체 코드 열람보다 결과를 만든 경로를 이해시키는 데 집중합니다."

### `A08`
- 20초:
  - "종범 자료 시연은 화면과 코드를 짝으로 설명합니다."
- 60초:
  - "Dify 워크플로와 웹 화면을 먼저 보여주고, 해당 결과를 만드는 코드 흐름을 간단히 연결합니다. 질문은 우리 팀에 바로 적용 가능한 포인트 중심으로 받겠습니다."

### `A09`
- 20초:
  - "마감 전에는 운영 리스크를 점검합니다."
- 60초:
  - "401/CORS/pending/period 오류 우회 시나리오를 빠르게 확인하고, 16:30 종료를 기준으로 Q&A 버퍼를 확보합니다. 시연 안정성을 최우선으로 마무리합니다."

### `R01`
- 20초:
  - "오늘 결과를 KPI 관점으로 정리합니다."
- 60초:
  - "SCM/CRM/BI 3축 기능 제안과 프로토타입 연결 가치를 확인했습니다. 다음 단계인 Agent/RAG/GraphRAG 확장을 준비할 수 있는 기반이 오늘 결과입니다."

### `R02`
- 20초:
  - "다음 회차 준비는 문서화가 핵심입니다."
- 60초:
  - "개발환경 표준, 모델 선택/비용 규칙, 실험 결과 기록을 팀 문서로 확정해야 재현성이 생깁니다. 오늘 내용은 실습 체험에서 운영 표준으로 넘어가는 출발점입니다."

### `R03`
- 20초:
  - "Q&A 후 액션아이템을 확정하고 종료하겠습니다."
- 60초:
  - "즉시 실행할 항목 3개를 합의하고, 강사/보조강사 역할 분담과 자료 전달 일정을 정리하겠습니다. 오늘 세션의 목표는 지식 전달보다 실행 가능한 상태를 만드는 것입니다."
