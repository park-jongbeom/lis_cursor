## 실행 환경 고정(중요)

이번 강의 리허설/시연은 **로컬호스트 직접 연동이 아니라 공인 URL 경로를 기본값으로 사용**합니다.

**동일 호스트 `https://lis.qk54r71z.freeddns.org` — 경로만 다르다**(서브도메인으로 규칙 사이트를 쪼개는 설계가 **아님**). 정본 표·“잘못된 방향” 제거는 **`docs/plans/lis_public_url_path_map.md`**.

**운영 표준(필독)**: 공인에서 **`/api/v1/`·`/docs`·`/ide/`** 등 IDR FastAPI 는 **§0 — 오직 로컬 우회(터널 등)**. `ga-server` 의 `idr-fastapi` 컨테이너를 공인 기본 업스트림으로 가정해 설명하지 않는다.

| 공인 경로 | 역할 | 업스트림(개념) |
|-----------|------|----------------|
| **`/`** | IDR Analytics **강의 데모** UI | nginx 정적 `demo/` 동기화본 |
| **`/apps`** 등 | **Dify** (Studio·앱) | Dify (예: Tailscale `:8080`) |
| **`/api/v1/`**, **`/docs`**, **`/ide/…`** | API·Swagger·교육생 규칙·ZIP | **동일 로컬 uvicorn**(터널로 엣지에 노출) — `demo/ide` 는 **로컬 리포** |

- API 베이스 URL: 데모 페이지의 API 입력 칸에서 공인 URL 기준으로 확인
- Bearer: 개발 우회 토큰 또는 로그인으로 발급한 JWT 사용

### 교육생 전용 · 규칙 요약 웹 페이지(강의 동선)

- **배포 목적(강조)**: 공인 `/ide/…` 는 **강사 화면만 보여주기**가 아니라, 교육생이 **직접 접속해 교육용 규칙 패키지를 다운로드·로컬에 적용**할 수 있게 하는 것이 목표다. ZIP·다운로드 경로·README 절차는 **`docs/plans/student_rules_download_lis_plan.md`** 에 정리한다.
- **목표 URL(공인)**: `https://lis.qk54r71z.freeddns.org/ide/docs/rules/`  
  - FastAPI가 리포의 `demo/ide` 를 마운트한 **업스트림 프로세스**가 살아 있어야 HTML·ZIP이 뜬다. 엣지 nginx(`ga-nginx`)는 `/ide/` 를 **데모(`/`)나 Dify(`/apps`)와 같은 호스트에서** 그 FastAPI로 넘기는 역할만 한다. **루트 데모가 된다고 `/ide`가 자동 완료되는 것은 아니다.**
- **공인 URL이 404·JSON(`{"detail":"Not Found"}`)일 때** (엣지 설정은 맞는데 앱에 정적 파일이 없는 경우가 흔함):
  1. **로컬 운영형 API**가 떠 있다면 동일 경로를 로컬로: `http://127.0.0.1:8010/ide/docs/rules/` (`infra/deploy/local-prod/README.md` 절차 6 — `.env.prod`·`IDE_STATIC_ROOT`·uvicorn 재시작).
  2. 또는 브라우저에서 리포 파일 직접: `demo/ide/docs/rules/index.html` (절대 경로로 `file://...` 열기 가능).
- **Cursor MCP(`user-ga-server-ssh`)**: **ga-nginx 컨테이너 안 설정 확인·(명시 시) 수정만**. 앱 컨테이너 배포·호스트에 파일 올리기로 이 페이지를 «고치려» 하지 않는다. 앱 반영은 리포·담당 배포 절차. (`docs/rules/project_context.md`, `infra/remote-proxy/README.md` 참고)

공인 URL에서 실제 확인 가능한 API 문서 경로(검증됨):

- Swagger UI: `https://lis.qk54r71z.freeddns.org/docs`
- OpenAPI JSON: `https://lis.qk54r71z.freeddns.org/openapi.json`

참고:

- `https://lis.qk54r71z.freeddns.org/api/v1/health` 는 현재 404일 수 있으므로, 헬스 체크는 데모 UI API 호출 성공 여부 또는 Swagger/OpenAPI 접근으로 대체해도 됩니다.

발표 자료의 화면 캡처/체크리스트/멘트도 위 공인 URL 기준으로 통일합니다. **다만 `/ide/docs/rules/` 캡처는 공인이 안 될 때 위 폴백(8010 또는 `index.html`)으로 동일 화면을 확보**하면 된다.

  - 문제 정의: “AI 산출물은 속도를 올리지만, 아키텍처/타입/성능/보안 요구조건까지 자동으로 만족시키진 못함”
  - 대응 전략: 문서(계약) + Gate(상태 머신) + 테스트(증적)를 결합
  - 결과: 변경 영향 범위가 줄고, 리뷰/운영 부담이 감소
  - 결론: AI 활용 그 자체보다 “AI를 통제 가능한 공정”으로 만드는 것이 핵심
- 발표 멘트(예시)
  - “AI를 많이 쓰는 게 아니라, AI가 만든 변경이 안전하게 반영되도록 공정을 설계한 프로젝트입니다.”
- 필요한 캡처: 없음

## 슬라이드 20. 문서 참조 순서(실제 세션 시작 루틴)

- 넣을 내용(순서 다이어그램)
  1. `docs/plans/plan.md`로 Phase/WBS 위치(무엇을 언제 끝낼지 계약) 확인
  2. `docs/CURRENT_WORK_SESSION.md`로 현재 Gate/승인 범위(상태 머신 입력) 고정

  4. 구현 중 `docs/rules/project_context`/`backend_architecture`/`dify_integration`로 아키텍처 계약 준수
  5. 완료 후 `docs/history/WORK_HISTORY.md`로 증적 이전(E2E 품질) 
- 발표 멘트(예시)
  - “세션 시작과 종료가 문서화되어, ‘내가 했던 결정’을 다음 사람이 그대로 재현할 수 있게 만들었습니다.”
- 필요한 캡처
  - `docs/` 하위 핵심 문서 리스트 캡처 1장

- 발표 멘트(예시)
  - “AI가 쓴 코드라도 최종 통과 기준은 사람이 아니라 도구가 판정합니다. 이게 팀의 일관성을 만듭니다.”

## 6) 추가 슬라이드용 발표 스크립트(18~26)

### 슬라이드 18 스크립트

“AI 개발에서 먼저 풀어야 하는 문제는 속도가 아니라 ‘변경이 안전하게 반영되도록 하는 통제’입니다. 이 프로젝트는 문서로 계약을 고정하고, Gate로 실행 권한을 쪼개며, 테스트로 증적을 남기는 방식으로 신뢰성을 확보했습니다.”

### 슬라이드 19 스크립트

“규칙은 계층으로 나눴습니다. 전역은 작업 문화(.cursorrules), 도메인은 아키텍처 계약(docs/rules/*), 운영은 품질과 보안 정책(.gitignore/커밋/비밀관리)으로 분리했습니다. 이 구조가 있어야 AI 변경도 예측 가능해집니다.”

### 슬라이드 20 스크립트

“세션 시작 루틴은 계약 확인 절차입니다. plan으로 마일스톤 위치를 고정하고, CURRENT로 Gate와 승인 범위를 고정합니다. 그리고 error_analysis로 과거 실패 모드를 먼저 봅니다. 구현 중에는 아키텍처 규칙을 계약처럼 적용하고, 종료 후에는 WORK_HISTORY로 증적을 이전합니다.”

### 슬라이드 21 스크립트

“승인 범위를 과대해석하는 실수가 가장 비쌉니다. 그래서 A는 설계/착수만, B는 구현 완료 후 강제 중지, C는 테스트 계획 승인 후 실행으로 단계 권한을 분리했습니다. 이게 리뷰 타이밍을 일정하게 고정합니다.”

### 슬라이드 22 스크립트

“프로젝트 스킬은 작업 순서를 고정하는 오케스트레이터입니다. 코드 생성보다 중요한 건 레이어 침범을 막고, 규칙/문서 정합성을 유지하는 프로세스입니다. 그래서 구현 전/중/후 체크포인트가 자동으로 유지됩니다.”

### 슬라이드 23 스크립트

“오류 분석 문서는 포스트모템 저장소입니다. AI로 발생한 실패를 오류 유형/상황/원인/재발 방지로 구조화해서 축적하고, 다음 세션 체크리스트로 재사용합니다. 같은 실패를 반복하지 않게 하는 게 목적입니다.”

### 슬라이드 24 스크립트

“품질은 도구로 객관화합니다. ruff와 mypy는 정적 방어, pytest는 실행 증적, pre-commit은 실패를 로컬에서 앞당깁니다. 그리고 staged/unstaged 혼재 커밋처럼 ‘롤백 비용’을 만드는 패턴 자체를 운영 규칙으로 막습니다.”

### 슬라이드 25 스크립트

“보안은 예외 없이 운영과 동일합니다. .env*를 전면 ignore하고, 템플릿은 점 없는 파일명으로 분리하며, 시크릿 스캔/푸시 보호 정책의 오탐 대응 원칙까지 같이 문서화했습니다.”

### 슬라이드 26 스크립트

“이 구조는 모델 변경에 무관한 팀 공정 템플릿입니다. 규칙 문서 세트, 세션 문서, Gate 승인 문화, 품질 자동화가 결합되어 있어서 다른 프로젝트에도 그대로 이식 가능합니다.”
