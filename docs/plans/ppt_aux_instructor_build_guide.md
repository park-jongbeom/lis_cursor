## 실행 환경 고정(중요)

이번 강의 리허설/시연은 **로컬호스트 직접 연동이 아니라 공인 URL 경로를 기본값으로 사용**합니다.

**동일 호스트 `https://lis.qk54r71z.freeddns.org` — 경로만 다르다**(서브도메인으로 규칙 사이트를 쪼개는 설계가 **아님**). 정본 표·“잘못된 방향” 제거는 **`docs/plans/lis_public_url_path_map.md`**.

**운영 표준(필독)**: 공인에서 **`/api/v1/`·`/docs`·`/ide/`** 등 IDR FastAPI 는 **`lis_public_url_path_map.md` 맨 앞 운영 원칙**대로 **오직 로컬 우회(터널 등)**. `ga-server` 의 `idr-fastapi` 컨테이너를 공인 기본 업스트림으로 가정해 설명하지 않는다.

| 공인 경로 | 역할 | 업스트림(개념) |
|-----------|------|----------------|
| **`/`** | IDR Analytics **강의 데모** UI | nginx 정적 `demo/` 동기화본 |
| **`/apps`** 등 | **Dify** (Studio·앱) | Dify (예: Tailscale `:8080`) |
| **`/api/v1/`**, **`/docs`**, **`/ide/…`** | API·Swagger·교육생 규칙·ZIP | **동일 로컬 uvicorn**(터널로 엣지에 노출) — `demo/ide` 는 **로컬 리포** |

- API 베이스 URL: 데모 페이지의 API 입력 칸에서 공인 URL 기준으로 확인
- Bearer: 개발 우회 토큰 또는 로그인으로 발급한 JWT 사용

### 교육생 전용 · 규칙 요약 웹 페이지(강의 동선)

- **배포 목적(강조)**: 공인 `/ide/…` 는 **강사 화면만 보여주기**가 아니라, 교육생이 **직접 접속해 교육용 규칙 패키지를 다운로드·로컬에 적용**할 수 있게 하는 것이 목표다. ZIP·다운로드 경로·README 절차는 **`docs/plans/student_rules_download_lis_plan.md`** 에 정리한다.
- **목표 URL(공인)**: `https://lis.qk54r71z.freeddns.org/ide/docs/rules/`  
  - FastAPI가 리포의 `demo/ide` 를 마운트한 **업스트림 프로세스**가 살아 있어야 HTML·ZIP이 뜬다. 엣지 nginx(`ga-nginx`)는 `/ide/` 를 **데모(`/`)나 Dify(`/apps`)와 같은 호스트에서** 그 FastAPI로 넘기는 역할만 한다. **루트 데모가 된다고 `/ide`가 자동 완료되는 것은 아니다.**
- **공인 URL이 404·JSON(`{"detail":"Not Found"}`)일 때** (엣지 설정은 맞는데 앱에 정적 파일이 없는 경우가 흔함):
  1. **로컬 운영형 API**가 떠 있다면 동일 경로를 로컬로: `http://127.0.0.1:8010/ide/docs/rules/` (`infra/deploy/local-prod/README.md` 절차 6 — `.env.prod`·`IDE_STATIC_ROOT`·uvicorn 재시작).
  2. 또는 브라우저에서 리포 파일 직접: `demo/ide/docs/rules/index.html` (절대 경로로 `file://...` 열기 가능).
- **Cursor MCP(`user-ga-server-ssh`)**: **ga-nginx 컨테이너 안 설정 확인·(명시 시) 수정만**. 앱 컨테이너 배포·호스트에 파일 올리기로 이 페이지를 «고치려» 하지 않는다. 앱 반영은 리포·담당 배포 절차. (`docs/rules/project_context.md`, `infra/remote-proxy/README.md` 참고)

공인 URL에서 실제 확인 가능한 API 문서 경로(검증됨):

- Swagger UI: `https://lis.qk54r71z.freeddns.org/docs`
- OpenAPI JSON: `https://lis.qk54r71z.freeddns.org/openapi.json`

참고:

- `https://lis.qk54r71z.freeddns.org/api/v1/health` 는 현재 404일 수 있으므로, 헬스 체크는 데모 UI API 호출 성공 여부 또는 Swagger/OpenAPI 접근으로 대체해도 됩니다.

발표 자료의 화면 캡처/체크리스트/멘트도 위 공인 URL 기준으로 통일합니다. **다만 `/ide/docs/rules/` 캡처는 공인이 안 될 때 위 폴백(8010 또는 `index.html`)으로 동일 화면을 확보**하면 된다.

### 핵심만 전달할 것 (말로 꼭 남겨야 하는 것)

1. **역할 나눔(슬라이드 1번 불릿)**: **최상위 계약**(`L1`·`.cursorrules`) / **도메인·워크플로 원문**(`L4`·`docs/rules/`) / **지금 작업 상태**(`plan`·`CURRENT`·`WORK_HISTORY`). 세 가지가 **다른 층**이라는 점만 말한다. L2·L3 파일명까지 줄줄 읽을 필요 없음.
2. **삼각형(슬라이드 2번 불릿)**: **문서 = 계약**, **Gate = 지금 어느 단계인지(상태 머신)**, **린트·테스트·훅 = 증적**. 제품 아키텍처(2-Tier, 엔드포인트 패턴 등)는 **이 슬라이드에서 다루지 않는다** — 웹 **6절(심화)** 도 열지 않는다.
3. **Gate 표 한 장(슬라이드 2번의 심장)**: A~E를 **한 줄씩 설명하지 말고**, “계획 승인 전엔 코드 금지 → 구현 끝나면 사람 확인 → 그다음에 테스트 계획” 흐름과 **표의 「승인 전 금지」 열**만 짚는다. Spring 트랜잭션 경계 비유는 여기까지.
4. **통제가 목표(슬라이드 3번 불릿)**: 속도가 아니라 **통제 가능한 공정**. 질문이 “왜 규칙이 많나?”로 가면 **짧게**: 긴 프롬프트만으로는 같은 실수가 반복되기 쉬워, **금지·게이트·`error_analysis` 누적**으로 옮겼다 — 필요 시 목차 `#c-shift` 또는 `#c-stories`에서 **카드 1장만** 보여 주고, 상세는 본 문서 말미 **「참고 — 지시→통제 전환」** 을 발표자가 읽는다.

## 실제 작성 덱 — 슬라이드 4. Gate A~E (승인 경계)

> **덱 번호 · 트랙 (정본)**  
> - **슬라이드 4** = 본 절(Gate A~E).  
> - 일일 대본 **`M04`** 후반에서 Gate 멘트를 연다(같은 `M04` 안에서 슬라이드 3 규칙 체계에 이어짐).  
> - **슬라이드 5 · Swagger·OpenAPI** = 트랙 **`M05`** — 아래 「슬라이드 5 — Swagger / OpenAPI」절.

### 슬라이드 초안 검토 (정본: `docs/rules/workflow_gates.md`)

초안:

- 제목: **Gate A~E로 승인 경계 분리**
- A: 구현 계획 승인  
- B: 구현 완료 후 강제 정지  
- C/D/E: 테스트 계획 / 실행 / 이력 전환  

| 항목 | 판단 | 권장 |
|------|------|------|
| **전체 구조** | 맞다. 구현 권한과 테스트 권한을 **승인 단계로 쪼개는** 요지가 정본 다이어그램과 같다. | 유지. |
| **A** | “구현 계획”만 있으면 **코딩 전인지·어디에 쓰는지**가 빠지기 쉽다. 정본은 **Gate A = `CURRENT`에 구현 상세 계획**을 쓰고 **사용자 승인 전 코드 변경 금지**다. | 슬라이드에 **「코딩 전」** 또는 **「`CURRENT`에 상세 계획」** 중 하나를 넣는 것이 좋다. (예: **A: 코딩 전 — 구현 상세 계획 승인**.) |
| **B** | “강제 정지”는 팀 용어로 **Gate B 직후 멈춤**(구현 끝 → 사용자 확인 → 그다음 C)과 잘 맞는다. | 유지. 발표에서 **테스트로 바로 가지 않는다**를 한 번 더 말한다. |
| **C** | 정본에서 C는 **테스트 실행 전** 단계다. “테스트 계획”만 있으면 **승인 후에만 D(실행)** 라는 점이 약해진다. | **C: 테스트 상세 계획 승인**(실행 전) 정도로 쓰면 AI 과대해석 방지 메시지와 맞는다. |
| **D** | “실행”에 **결과 기록**이 포함된다는 점을 슬라이드에 쓸 공간이 있으면 좋다. | 예: **D: 테스트 실행·검증 결과 기록**. |
| **E** | “이력 전환”은 맞다. 정본은 `WORK_HISTORY` + `plan.md` 갱신 + **다음 세션용 `CURRENT`로 교체**까지 포함한다. | 필요 시 괄호로 **「다음 세션 문서」** 한 단어 추가. |

**한 줄 정본 출처** (발표자·슬라이드 각주용): `workflow_gates.md` 요약 다이어그램 — A·B·C 각각 **사용자 승인(⏸)** 이 있고, **「Gate A 승인 = 구현 착수만」** 이 테스트까지 포함하지 않는다는 점이 중요하다.

### 왜 슬라이드 3에서 Gate를 짚은 뒤, 슬라이드 4에서 **다시** 설명하는가

슬라이드 3(규칙 웹·브라우저)에서는 **의도적으로** Gate를 **표 한 장·흐름 몇 마디**로만 끝낸다. 같은 날 오전 흐름에서 Gate를 두 번 다루는 것은 중복이 아니라 **역할 분담**이다.

1. **인지 부담 분리**  
   슬라이드 3은 **레이어(L1/L4/L5)·세션 문서·ZIP·통제 목표**까지 한꺼번에 올라간다. 여기서 A~E를 **글자 하나하나**까지 풀면 청중은 “규칙 시스템 전체”와 “승인 상태 머신”이 한 덩어리로 뭉개져 기억하기 어렵다. 그래서 3에서는 **「금지 열이 있고, 계획→구현→확인→테스트가 끊긴다」** 수준만 남기고 끝낸다.

2. **두 번째 슬라이드 = 추상 → 운영 규칙**  
   3번은 **지도(어디에 뭐가 있는지)** 이고, 4번은 **운전 매뉴얼(어떤 순서로 손대면 안 되는지)** 에 가깝다. 특히 **B에서 멈춤**, **A 승인이 테스트까지가 아님**은 AI·사람 모두 **자주 깨는 지점**이라(`error_analysis`, `workflow_gates.md` 「승인 범위 해석」), 슬라이드 한 장을 **통째로 할애**하는 것이 교육 효과 대비 비용이 낮다.

3. **오전 스크립트(M04 → M05)와의 정렬**  
   **`M04`**: 규칙 체계(슬라이드 3)에 이어 **Gate**(슬라이드 4) 멘트. **`M05`**: **슬라이드 5 Swagger·OpenAPI**. 덱에서 **3번**에 Gate 표를 잠깐 보여 준 것은 4번 **승인 경계**의 예고로 쓴다.

4. **현장 질문 대비**  
   “진행해라 한 마디로 테스트까지 돌린 적 있다”는 식의 질문은 3번 맥락만으로는 답이 길어진다. 4번에서 **A/B/C에 각각 승인이 있다**는 것을 한 번 더 박아 두면, 오후 실습·코드 시연으로 넘어갈 때 **프로세스 논쟁**이 줄어든다.

5. **반복의 법칙**  
   교육에서 **핵심 안전 규칙**은 “한 번에 크게 + 나중에 같은 주제만 좁혀서” 두 번 말하는 편이 유지된다. Gate는 **품질·권한 사고**와 직결되므로 의도적 반복으로 본다.

**정리 (발표자 한 줄)**: “아까 웹에서는 Gate가 **전체 그림 속 한 조각**이었고, 지금 슬라이드는 **승인만 잘라서** 보여 드립니다.”

### 슬라이드 4 — 전달 순서·시간

- **1분(압축)**: 제목 + 불릿 읽기 + “A 승인은 구현만, B 끝나면 멈춤, 테스트는 C 이후” 한 문장.  
- **2~3분(권장)**: 위 + **승인 과대해석 금지**(정본 한 줄: A 승인 ≠ C/D 자동 승인) + 질문 1회 받기.  
- **시각 자료**: 가능하면 `workflow_gates.md`의 **텍스트 다이어그램**을 슬라이드에 옮기거나, 규칙 웹의 Gate 표(`#c-gates`)를 **슬라이드 4에서만** 다시 띄워 **같은 표를 두 번째 관점**으로 읽는다(“이번에는 열보다 **멈춤 지점**만”).

### 캡처·자료

- 규칙 웹 Gate 표를 4번에서 재사용할 경우: 슬라이드 3과 **동일 화면**이어도 되며, 강조 박스만 B·C 사이에 그려도 된다.

## 실제 작성 덱 — 슬라이드 5. Swagger / OpenAPI — API 계약 (FastAPI 자동 스펙)

> **덱 · 트랙 (정본 — 사용자 덱 기준)**  
> - **슬라이드 제목 예**: `Swagger·OpenAPI — API 계약 (FastAPI 자동 스펙)`  
> - **슬라이드 번호**: **5**  
> - **오전 대본 ID**: **`M05`** (`lecture_talk_track_2026_03_28.md`의 `### M05`)  
> - **직전 슬라이드**: **4** (Gate A~E)  
> - **URL**: `https://lis.qk54r71z.freeddns.org/docs` · `https://lis.qk54r71z.freeddns.org/openapi.json`

### 목적에 대한 판단

**“AI로 프로젝트를 구축하면서 Swagger까지 자동으로 나온 것을 보여준다”**는 방향은 **맞다.** 이 저장소는 **FastAPI**라서, 엔드포인트·Pydantic 스키마·태그 등이 잡히면 **`/docs`·`/openapi.json`이 코드에서 파생**된다. 즉 **별도 스펙 문서를 사람이 매번 맞춰 쓰지 않아도** 계약 표면이 생긴다는 점을 오전 마무리로 쓰기 좋다.

**정확한 한 줄**: “AI가 **OpenAPI YAML을 손으로 다 쓴** 것”이 아니라, **AI·사람이 쓴 라우트/모델 위에 FastAPI가 스펙을 생성**한다. 그래도 청중 입장에서는 **“코드 기반으로 스웨거가 따라왔다”**는 인상 목적은 달성된다.

**Swagger UI에 보이는 “설명”은 어디서 오나**  
화면에 읽히는 **요약·설명·파라미터 설명**은 프레임워크가 공중에서 만드는 게 아니라, **코드 안 메타데이터**에서 온다. 예: 엔드포인트 **`summary` / docstring**, Pydantic **`Field(description=...)`**, 라우터 **`tags`**, **`response_model`** 등. 이걸 비워 두면 경로·타입만 나오고 칸이 허전해진다.

**이 프로젝트 서술 보강(권장)**  
위 메타데이터는 **사람이 전부 손으로만 채운 것도, 전부 자동도 아니다.** 실제 흐름은 **AI가 라우트·스키마와 함께 설명 문구 초안을 같이 쓰고**, 규칙·리뷰·Gate·린트로 다듬은 뒤 코드에 남기므로 **스웨거에도 그대로 반영**된다고 말하면, “FastAPI가 알아서 다 해준다” 오해와 “설명은 따로 문서 팀이 써야 한다” 오해를 **동시에** 줄일 수 있다.

**트랙 ID**: 본 슬라이드는 일일 대본 **`M05`** 와 **덱 슬라이드 5**에 고정한다(이전에 문서에 적혀 있던 M07 표기는 폐기).

### 슬라이드에 두 URL 외로 **첨언하면 좋은 것**

| 첨언 | 이유 |
|------|------|
| **FastAPI → OpenAPI 자동 생성** (한 줄) | 경로·메서드·스키마 **뼈대**가 코드에서 파생된다는 프레임. |
| **설명 문구는 코드 메타데이터** — docstring · `summary` · `Field(description)` · 태그 등; **비우면 스웨거가 허전** | “짚을 부분이 스웨거 설명 아닌가?”에 대한 **정직한 답**. |
| **이 리포에서는 그 메타데이터도 AI 보조로 함께 작성** → 스웨거 가독성까지 같은 파이프라인 | 사용자 제안 반영: **자동 생성 + AI가 설명까지 같이 채운 사례**. |
| **태그로 도메인 묶음** — `auth` · `datasets` · `scm` · `crm` · `bi` · `agent` | 오후 SCM/CRM/BI 데모와 **같은 말**을 미리 깔아 둠 (`app/main.py` `openapi_tags`·라우터 구조와 일치). |
| **오후 연결 한 줄** — “데모 UI·Dify는 이 `/api/v1/` 계약을 부른다” | 규칙(Gate) 다음에 **실제 산출물**로 내려감. |
| (선택) **`compact=true`** | 이 프로젝트 규칙상 LLM·Dify용 요약 쿼리; 한 불릿 또는 발표자 노트만. |
| (발표자 노트) 최종 문구는 **사람이 승인·수정**한 결과물 — Gate·리뷰와 연결 가능 | 과장 방지. |

### 하지 말 것

- 엔드포인트를 **하나하나 판서**하지 않는다. **태그 펼쳐서 구조만** 30초~1분.  
- 공인이 안 뜨면 **`실행 환경 고정`** 절의 폴백(로컬 `/docs` 등)으로 동일 메시지만 전달.

### 데모 시간

- **1~2분**: 브라우저 `/docs` → 태그 2~3개만 펼침 → `openapi.json`은 “머신이 읽는 동일 계약” 한 마디.  
- **리허설**: 네트워크·로그인 필요 여부만 확인(Swagger는 보통 공개, API 호출은 JWT 별도).

### 스토리라인·전제: 도메인 비전문 · AI 보조 단기 구축 · 향후 도메인 규칙으로 성숙

이 데모 저장소를 소개할 때 **“SCM/CRM/BI 업무를 수십 년 해 온 도메인 전문가가 설계한 완성품”**처럼 말하면 기대치가 어긋난다. 강사가 **의도적으로** 쓰기 좋은 전제는 다음과 같다.

**1) 사실 관계(강조해도 되는 문장)**  

- 본인은 이 데모가 다루는 **세부 도메인(예: 해당 업종 KPI·현장 의사결정 규칙)** 에 대해 **전문가 수준의 지식을 갖고 시작하지 않았다.**  
- 코드베이스·API·샘플 데이터·화면 흐름은 **AI 도구를 활용해 시스템을 분석·설계하고**, 짧은 기간(**총 약 3~4일** 정도의 오더) 안에 **수직 슬라이스(업로드 → Tier1 분석 → Tier2 요약 연결)** 를 **동작 가능한 형태**로 올린 사례다.  
- 따라서 오늘 보여 주는 것은 **“현장에 그대로 납품할 최종 솔루션”**이라기보다, **교육·시연·계약(API)·통제 축(Gate·규칙 문서)이 같이 깔린 1차 버전**에 가깝다.

**2) 왜 솔직한 전제가 유리한가**  

- 청중(실무 개발자)은 **“AI로 빠르게 만들었다”**는 말만 들으면 **품질·책임**을 의심하기 쉽다. **도메인을 깊이 모른 채 시작했다**고 말하면 오히려 **과장이 아닌 투명성**으로 읽힌다.  
- 동시에 **오전에 이미 설명한** `.cursorrules`·`docs/rules`·Gate·테스트와 연결하면, “빠른 구축 = 아무 규칙 없이”가 아니라 **“짧은 시간 안에도 계약·승인·증적 축을 같이 깔았다”**는 메시지가 된다.  
- **속도만** 강조하지 말고 반드시 **한 짝**을 붙인다: “그만큼 **다음 레이어**에서 **실제 도메인**으로 규칙과 지표를 다시 잡고 솔루션을 보강할 예정이다.”

**3) “3~4일”이 의미하는 것·의미하지 않는 것**  

| 의미해도 되는 해석 | 피해야 하는 오해 |
|-------------------|------------------|
| **교육용 수직 슬라이스**를 API·UI·LLM 연동까지 **한 흐름**으로 만든 **캘린더 기준 기간** | “업계 표준 컨설팅 산출물 전부 완료” |
| **샘플 CSV·기본 컬럼 가정** 위에서 **파이프라인·계약**을 증명 | “특정 고객사 실데이터·법규·승인 프로세스까지 반영 완료” |
| AI가 **초안·반복 작업**을 많이 맡고, 사람이 **방향·승인·문서·린트**로 다듬은 기간 | “버튼 한 번으로 전부 자동 완성” |

**4) AI가 한 일 vs 사람·저장소가 한 일 (발표에서 짧게 나누는 법)**  

- **AI 쪽**: 요구를 질문으로 쪼개기, 라우트·스키마·데모 HTML·Dify 연동 초안, 문서 초안, 반복 수정.  
- **사람·저장소 쪽**: `plan`/`CURRENT`·Gate로 **승인 경계**, `docs/rules`로 **아키텍처 계약**, ruff·mypy·pre-commit·테스트로 **증적**, `error_analysis`로 **실패 모드 축적**.  
- 한 줄 요약 예: “**뼈대와 초안의 속도는 AI가**, **무엇을 하면 안 되는지와 언제 멈출지는 저장소가** 잡는다.”

**5) 다음 단계(로드맵) — 강사가 말하기 좋은 ‘올바른 성숙 방향’**  

- **실제 도메인**에서: 용어 정의, KPI·임계값, 데이터 품질 규칙, “이 지표로 무슨 결정을 하는가”를 **원천**으로 삼는다.  
- 그에 맞춰 **핵심 규칙을 재정립**한다: 어떤 열이 필수인지, `compact`에 무엇을 넣을지, 어떤 분석은 Tier1에 남기고 어떤 해석은 Tier2에 둘지, Dify 프롬프트·검증 시나리오.  
- 그다음 **솔루션을 보강**한다: 스키마·엔드포인트·잡 파라미터·샘플이 아닌 **운영 데이터** 경로, 모니터링, 접근 통제.  
- 이 순서를 말하면 **“AI로 때운 데모”**가 아니라 **“1차 계약을 깐 뒤 도메인으로 정식화한다”**는 엔지니어링 서사가 된다.

**6) 슬라이드에 넣을 수 있는 선택 불릿 (슬라이드 6에 한 줄 추가할 때)**  

- (선택) “도메인 전문가로 시작하지 않았고, **AI 보조로 약 3~4일** 안에 **동작하는 데모 축**을 올린 사례”  
- (선택) “다음은 **현장 도메인으로 규칙·지표를 재정의**하고 그에 맞게 **보강**하는 단계”

**7) 발표 시간별 멘트 예시 (발표자 노트에 그대로 써도 됨)**  

- **20초**: “이 데모는 해당 도메인을 깊게 알고 시작한 프로젝트는 아니고, AI로 시스템을 분석·구축하는 데 도움을 받아 며칠 안에 화면·API·요약까지 연결한 1차 버전입니다. 다음은 실제 도메인으로 규칙을 다시 잡고 솔루션을 보강하는 방향이 맞다고 봅니다.”

### 1) “데모 접속”이 가리키는 화면

- **구현물**: 단일 페이지 **`demo/index.html`** — FastAPI가 **공인·로컬 모두**에서 같은 파일을 서빙할 수 있다(`demo/README.md`, `lis_public_url_path_map.md` §0).  
- **시스템 흐름 그림**: 제목·규칙 링크 바로 아래 **`#demo-flow-overview`** 패널에 **2-Tier**(브라우저 → Tier1 FastAPI/Pandas → Tier2 Dify/LLM)와 **complexity 라우팅** 개념을 **SVG 다이어그램**으로 넣어 두었음 — 슬라이드 없이 **브라우저만** 확대해 짚어도 됨.  
- **공인 강의 URL(예)**: 사이트 루트 **`/`** 또는 **`/index.html`** — 브라우저 제목은 **「IDR Analytics 데모」**, 상단 부제는 개발/운영 시 Bearer·로그인 안내 한 줄.  
- **교육생 링크**: 제목 아래·하단에 **`/ide/docs/rules/`** (규칙 가이드) — 이 슬라이드에서는 **데모 동선만** 말하고, 규칙 웹은 오전에 이미 다뤘으면 짚지 않아도 된다.  
- **로컬**: VS Code Live Server 등으로 `demo/index.html`을 띄우되, 스크립트가 **API 베이스를 `http://127.0.0.1:8000/api/v1`** 으로 잡는 경우가 많다(포트 5500·5173 등). **HTML을 연 호스트와 API 호스트가 다를 수 있다**는 점을 시연 전에 한 번 확인한다.  
- **로그인 UI**: HTML에서 로그인·API 베이스·Bearer 입력 행은 **`demo-hidden`** 이라 **강의 빌드에서는 숨겨진 채** 쓰는 경우가 많다. 대신 **`DEMO_AUTO_LOGIN`** 이 켜져 있으면 페이지 로드 직후 **`/auth/login`** 으로 JWT를 받아 **`/datasets/sample-catalog`** 등 후속 호출이 동작한다. **자동 로그인이 실패하면** 빈 화면처럼 보이지 않더라도 샘플 목록이 안 뜨므로, 리허설에서 반드시 **`dataset_id` 줄까지** 확인한다.

### 2) CSV 업로드 → `dataset_id` 확보 (첫 번째 패널)

화면 **맨 위 큰 패널**이 전부다. 순서는 슬라이드 불릿과 동일하게 설명하면 된다.

| UI 요소 | 동작·의미 |
|--------|-----------|
| **CSV 파일** + **데이터셋 이름** + **`dataset_type`** (`scm` / `crm` / `bi` / `mixed`) + **업로드** | `POST /api/v1/datasets/upload` — 응답의 **`dataset_id`**(UUID)가 아래 줄에 **녹색 굵게** 표시된다. 업로드 실패 시 `(실패)` 및 첫 패널 하단 인접 **상태 영역**에 에러 문구. |
| **내장 샘플 선택** + **선택 샘플 즉시 업로드** | `GET /datasets/sample-catalog`로 채워진 셀렉트 — `POST /datasets/upload-sample` 로 서버에 있는 **`demo/sample_data/*.csv`** 를 바로 올린다. **강의 대기시간 줄이기에 유리**. |
| 샘플 선택 시 **회색 점선 박스** | 선택 파일의 **컬럼 칩** + 상위 3행 미리보기 테이블 — “이 CSV가 SCM인지 CRM인지”를 말로 설명하기 좋다. |
| **`현재 dataset_id:`** | 모든 SCM·CRM·BI·에이전트 호출의 **공통 키**. 비어 있으면 탭에서 버튼을 눌렀을 때 **“먼저 업로드”** 류의 오류가 난다. |

**발표 한 줄**: “한 번 올린 `dataset_id`로 뒤 탭들이 **같은 업로드 결과**를 공유한다.”

### 3) SCM / CRM / BI 탭 — 화면에 실제로 있는 버튼과 산출물

탭 바는 **네 칸**이다. 이름은 데모 HTML과 **글자까지 맞출 것**: **「SCM 수요예측」**, **「CRM 이탈·클러스터」**, **「BI 트렌드」**, **「Dify AI 요약」**(마지막 칸은 다음 소절).

**SCM 수요예측** (`#panel-scm`)

- 입력: `target_column`, `date_column`, `group_by`, `test_codes`, `forecast_days` — 기본값은 **`scm_sample.csv`** 에 맞춰져 있다.  
- **예측 실행 → 폴링**: `POST /scm/forecast` 후 **`/scm/forecast/{job_id}`** 를 최대 약 180초 폴링. 아래 **라인 차트**(Chart.js)에 상위 시리즈 예측이 그려진다.  
- **상태 줄**(`#statusScm`): 잡 ID, 완료 시 모델명·시리즈 개수 등.  
- **주의**: **ARQ 워커**가 없으면 잡이 **`pending`** 에서 멈출 수 있다 — `demo/DEMO_SCRIPT.md` 리허설 체크리스트와 동일.

**CRM 이탈·클러스터** (`#panel-crm`)

- **이탈 위험 분석**: 동기 **`GET /crm/churn-risk`** — 바 차트 + **코드/이름/경과일/위험/세그먼트** 테이블.  
- **클러스터 실행 → 폴링**: `POST /crm/cluster` + **`/crm/cluster/{job_id}`** — 차트·표가 **클러스터 결과**로 갱신됨. 여기도 **워커 필요**.  
- 두 버튼 모두 **같은 캔버스·표 영역**을 쓰므로, 시연 순서를 정해 두면 청중이 덜 헷갈린다(예: 이탈만 또는 클러스터만).

**BI 트렌드** (`#panel-bi`)

- **지역 히트맵**: `GET /bi/regional-heatmap` — **`period`는 CSV의 기간 값과 문자열이 정확히 같아야** 한다(샘플 기본 `2024-01`). 표 + 막대 차트, 인사이트 문장.  
- **YoY 비교**: `GET /bi/yoy-comparison` — 연도별 YoY 막대 차트.  
- 둘 다 **동기 호출**이라 워커 없이도 동작하기 쉬워 **시연 폴백**으로 좋다.

### 4) “Dify 워크플로로 요약/해석 연결” — 이 프로젝트에서의 두 층

슬라이드 불릿은 **교육 계획(Dify Studio 워크플로)** 과 **이 HTML의 네 번째 탭**을 한 줄에 묶은 것이다. 강사는 둘을 구분해 말해 주는 것이 좋다.

**A. 데모 HTML 안: 「Dify AI 요약」 탭** (`#panel-agent`)

- **질의** 입력란(`#agentQuery`) + **에이전트 실행**. 기본 placeholder와 유사한 문장 외에, 시연용 복사 문장은 위 **「6-3)」질의 리스트** 참고.  
- **`POST /api/v1/agent/query`** — 응답 **`answer`** 가 큰 **textarea**에 채워지고, 상태 줄에 **`route_used`**, **`session_id`** 가 찍힌다.  
- **의미**: 브라우저는 **여전히 Tier1 같은 호스트**만 치고, **Tier2(Dify)는 백엔드가 대신 호출**하는 형태 — 오전에 말한 **2-Tier**와 연결 가능.  
- 타임아웃 **최대 3분** — 네트워크·Dify 기동 여부에 따라 **수십 초** 걸릴 수 있음을 미리 알려 두면 좋다. 502 시 메시지에 **Dify/역프록시** 힌트가 붙는 경우가 있다(HTML `fmtErr`).

**B. Dify Studio (`localhost:8080` 등) — 워크플로 캔버스**

- `demo/DEMO_SCRIPT.md` **4번**: HTTP 노드로 CRM·BI 등을 붙이고 LLM까지 가는 **블록 다이어그램** 시연.  
- 슬라이드 6 한 장에서는 **“같은 백엔드를 HTML 버튼에서도, Studio 파이프라인에서도 부를 수 있다”** 수준으로만 막고, **캔버스를 노드별로 짚는 상세**는 아래 **「B-1」** 을 따른다.

#### B-1) 앱 **「IDR CRM·BI Tier2」** 워크플로 — DSL 기준 시연 가이드

강의에서 **Dify에 로그인 → 해당 앱(워크플로) → 실행/미리보기**로 들어가 **캔버스를 왼쪽에서 오른쪽으로** 짚을 때 쓴다. 아래는 **보낸 DSL** 구조를 기준으로 한 **노드·입력·주의** 정리다. (DSL에 **JWT·URL** 이 박혀 있을 수 있으니, **저장소·공유 문서에는 비밀을 붙이지 말 것** — 노드에는 **`make dify-fastapi-jwt-bearer`** 등으로 받은 값을 **강의 전에만** 넣고, 유출 시 **토큰 폐기·재발급**.)

**앱 메타 (DSL)**

| 항목 | 값 |
|------|-----|
| **이름** | `IDR CRM·BI Tier2` |
| **모드** | `workflow` |
| **설명(앱)** | FastAPI **`compact`** 엔드포인트 **2건** 호출 후 **Ollama** 로 한국어 인사이트 생성 (IDR Phase 6) |
| **Tier2 플러그인** | Marketplace **`langgenius/ollama`** (워크플로 LLM 노드에서 Ollama 제공자로 사용) |
| **의도** | **Tier1**이 만든 JSON을 **짧게(`compact=true`)** 받아 **한 번에 LLM 컨텍스트**로 넘기고, **사용자 질문**과 합쳐 답을 쓴다. |

**그래프 흐름 (실행 순서)**

노드는 **직렬**이다. **병렬 분기 없음**.

```text
[시작] → [HTTP CRM churn-risk] → [HTTP BI regional-heatmap] → [변수 집계자] → [LLM (Ollama)] → [끝]
```

- **의미**: 먼저 **CRM compact JSON**을 받고, 이어서 **BI compact JSON**을 받은 뒤, **집계 노드**가 두 HTTP **응답 body**를 합쳐 문자열로 만든다. 그 문자열과 **시작 노드의 `user_query`** 를 **LLM**에 넣어 최종 텍스트를 낸다.

**1) 시작 노드 (`start`)**

| 변수 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `user_query` | paragraph | 예 | 최종 LLM에 같이 넘어가는 **자연어 질문** (에이전트·앱 입력과 맞출 수 있음). |
| `dataset_id` | text | 예 | Tier1 **`GET /crm/churn-risk`**, **`GET /bi/regional-heatmap`** 에 공통으로 쓰는 UUID. **데모 HTML의 녹색 `dataset_id`** 와 **동일 값**을 복사해 넣는다. |
| `period` | text | 예 | **BI 히트맵** 쿼리용. **CSV `period` 열 값과 문자열이 정확히 같아야** 한다 (예: 샘플 `bi_sample.csv` 계열이면 `2024-01`). 노드 설명에도 “regional-heatmap 필수”라고 적혀 있는 맥락과 동일. |

**시연 멘트 예**: “시작 노드에 **API가 요구하는 키만** 박아 두었습니다. `dataset_id`는 아까 업로드한 것과 같고, `period`는 BI 샘플과 **글자 하나까지** 맞춰야 합니다.”

**2) HTTP 노드 ① — `CRM churn-risk`**

- **메서드**: `GET`  
- **경로 개념**: `/api/v1/crm/churn-risk`  
- **쿼리 (DSL 패턴)**: `dataset_id={{시작.dataset_id}}`, **`compact=true`**, `top_n=10` (또는 DSL에 맞춘 값).  
- **인증**: 헤더 **`Authorization: Bearer <JWT>`** — **Dify UI에서만 설정**. 문서·채팅·DSL Git 커밋에 **실토큰을 넣지 말 것**.  
- **역할**: **이탈 위험 상위 N** 등 **CRM 요약 JSON**을 Tier1에서 가져온다 — **계산은 전부 FastAPI**.

**3) HTTP 노드 ② — `BI regional-heatmap`**

- **메서드**: `GET`  
- **경로 개념**: `/api/v1/bi/regional-heatmap`  
- **쿼리 (DSL 패턴)**: `dataset_id={{시작.dataset_id}}`, **`period={{시작.period}}`**, **`compact=true`**.  
- **인증**: 동일하게 **Bearer JWT**.  
- **역할**: **지역·기간** 관점 **BI 요약 JSON**을 가져온다. **`period` 불일치 시 404/에러**가 나기 쉬움 — 리허설 필수.

**4) 변수 집계자 (`variable-aggregator`)**

- **입력**: 위 **두 HTTP 노드의 `body`** (응답 본문).  
- **출력**: 하나의 **문자열**(`output`)로 이어 붙인 뒤 **LLM 노드**로 전달.  
- **시연 멘트 예**: “여기서 **두 번의 Tier1 호출 결과**를 **한 덩어리 텍스트**로 합칩니다. LLM은 **원시 CSV를 안 보고**, **이미 압축된 JSON 문자열**만 봅니다 — 그래서 `compact=true`가 규칙으로 박혀 있습니다.”

**5) LLM 노드 — `LLM (Ollama)`**

- **제공자 (DSL)**: `langgenius/ollama/ollama` 플러그인.  
- **모델명 (DSL 예시)**: `alibayram/Qwen3-30B-A3B-Instruct-2507:latest` — **강의 PC Ollama에 실제로 pull·등록된 이름**과 맞춰야 한다. (다른 모델을 쓰면 Dify에서 **모델 선택만** 바꾸면 됨.)  
- **옵션 (DSL)**: `temperature: 0.3`, 모드 `chat`.  
- **시스템 프롬프트 (DSL 요지)**: “당신은 **의료 검사 영업 분석** 전문가입니다. 주어진 JSON을 바탕으로 **한국어**로 실무적 인사이트를 제시하세요. **수치를 지어내지 말고**, 과장한 해석은 피하세요.”  
- **유저 프롬프트 구조 (DSL 요지)**  
  - `데이터 (연결된 compact JSON):` + **`{{변수 집계자.output}}`**  
  - `사용자 질문:` + **`{{시작.user_query}}`**  
  - “위 데이터와 질문에 대해 **한국어로** 답변하세요.”  

**시연 멘트 예**: “**숫자는 JSON에만 있고**, 모델은 **그 JSON과 질문**만 받습니다. 시스템 역할은 **도메인 톤**(여기서는 의료 검사 영업)을 잡아 준 것이고, **강의 목적이면** 나중에 일반 SCM/CRM 문구로 바꿀 수 있습니다.”

**6) 끝 노드 (`end`)**

- **출력 변수**: LLM 노드의 **`text`** → 앱 실행 화면에 **최종 답변**으로 표시.

**리허설 체크리스트 (Studio 진입 시)**

1. FastAPI·공인/로컬 URL이 Dify HTTP 노드와 **동일 베이스**인지 확인.  
2. **JWT** 갱신 후 두 HTTP 노드 헤더에 반영 (`DEMO_SCRIPT.md`·`make dify-fastapi-jwt-bearer`).  
3. 데모에서 **`dataset_id`** 복사 → Dify 실행 입력에 붙여넣기.  
4. **`period`** = 해당 데이터셋 CSV의 기간 값과 **완전 일치**.  
5. **한 `dataset_id`로 CRM·BI API가 둘 다 유의미한 JSON을 주는지** 미리 확인 — `crm` 전용·`bi` 전용 CSV만 올렸다면 **한쪽 HTTP는 에러·빈 응답**이 될 수 있음. 강의용으로는 **두 엔드포인트가 모두 성공하는 업로드 조합**을 리허설에서 고정하는 것이 안전하다.  
6. Ollama 기동·모델 태그가 LLM 노드 설정과 **일치**.  
7. **Publish** 된 워크플로/앱인지 확인 후 시연.

**슬라이드·말하기 연결**

- **슬라이드 7**에서 말한 **“Dify = 오케스트레이션, Ollama = 추론”** 을 이 캔버스에 대응시키면: **HTTP·집계 = Dify가 짠 파이프**, **LLM 블록 = Ollama에 붙은 모델**.  
- **데모 HTML `agent/query`** 와 **같은 Tier1 계약**을 쓰되, Studio 버전은 **노드가 보이므로** “교육용 투명성”이 좋다고 한 줄 말하면 된다.

### 5) 슬라이드에 넣기 좋은 캡처·리허설 체크

- **한 장 요약용 스크린샷**: 가능하면 **첫 패널 + 탭 네 칸이 한 화면에 들어가는 폭**으로 찍거나, 상단 패널과 탭 바만 크롭.  
- **리허설** (`demo/DEMO_SCRIPT.md`와 동일 계열): health·로그인(또는 자동 로그인)·**샘플 업로드 한 번**·SCM 또는 BI **한 경로 성공**·가능하면 **에이전트 탭 1회**.  
- **시간이 촉박할 때**: 샘플 즉시 업로드 → **CRM 이탈**(동기) 또는 **BI 히트맵** → (시간 되면) 에이전트 한 줄.

### 6) 흐름 다이어그램(`#demo-flow-overview`) 시연 활용법 · **「Dify AI 요약」탭 질의 리스트**

데모 HTML 상단 **2-Tier·complexity** 그림은 **슬라이드 없이** 구조를 고정할 때 쓴다(**6-1**).  
여기서 말하는 **질의**는 교육생이 강사에게 하는 질문이 **아니라**, 강사가 데모 화면 **「Dify AI 요약」** 탭의 **질의 입력란**(`#agentQuery`)에 붙여 넣어 **`에이전트 실행`** 할 **자연어 프롬프트** 목록이다. 발표자 노트·리허설에 복사해 쓴다.

#### 6-1) 권장 시연 시퀀스(다이어그램을 “축”으로 쓰기)

| 단계 | 하는 일 | 효과 |
|------|---------|------|
| 0 | 페이지 열자마자 **`#demo-flow-overview`** 가 보이게 스크롤 위치 유지(또는 맨 위) | “한 페이지 안에 **아키텍처 + 조작 UI**가 같이 있다”는 인상 |
| 1 | 박스 **왼쪽→가운데**만 손가락/포인터로 짚음: 브라우저 → HTTPS `/api/v1` → Tier1 | 업로드·탭을 누르기 **전에** 기대치 고정 |
| 2 | **아래쪽** `complexity 기준 라우팅` 박스 읽기(또는 요약) | “왜 탭이 나뉘는지”를 **슬라이드 없이** 설명 |
| 3 | **오른쪽 Tier2** 짚은 뒤 “지금은 먼저 Tier1만”이라고 하고 **샘플 업로드**로 이동 | 시연 팁(가이드 하단 문구)과 **동일 동선** |
| 4 | SCM/CRM/BI 중 하나 실행 후, 필요할 때만 **「Dify AI 요약」** 탭 | 그림의 **화살표 순서**와 실제 클릭이 맞는다고 말할 수 있음 |

#### 6-2) 질의 입력 전에(선행 조건·주의)

- **`dataset_id` 필수**: 업로드·샘플 즉시 업로드 후 **녹색 ID**가 잡힌 뒤 탭으로 이동한다.  
- **응답 품질과 `dataset_type`**: 한 CSV만 올렸다면 **해당 도메인 위주**로 질문하는 것이 안전하다. `mixed` 등 **한 데이터셋에 여러 성격이 섞인 경우**에만 “SCM·CRM·BI를 한꺼번에” 요약하는 식의 질의가 잘 맞는다. **샘플만 빠르게 돌릴 때**는 아래 **A(기본)** 또는 **D~F**처럼 **범위를 좁힌 질의**를 권장한다.  
- **시간**: Tier2는 **수십 초~최대 3분** 걸릴 수 있다. 강의에서는 **질의 1개만** 실행하고 결과를 읽는 편이 낫다.  
- **실패 시**: 502·타임아웃이면 **Tier1 탭 성공**만으로도 시연은 성립한다 — `demo/DEMO_SCRIPT.md` 문제 표 참고.

#### 6-3) 「Dify AI 요약」탭에 넣을 **질의 문장 리스트**(복사용)

아래는 **한국어** 기준 예시다. 필요하면 길이만 줄여 쓴다.

| ID | 용도 | 질의 문장(입력란에 그대로) |
|----|------|---------------------------|
| **A** | **기본·복합**(HTML 기본값과 동일 계열) | 이 데이터셋의 SCM·CRM·BI 관점에서 핵심 인사이트를 한국어로 요약해 줘. |
| **A′** | A의 짧은 버전(시간 절약) | 이 데이터셋 핵심만 한국어로 세 줄로 요약해 줘. |
| **B** | 경영 브리핑 톤 | 이 데이터셋 분석 결과를 경영진 보고용으로, 한 단락(5~7문장) 한국어 브리핑으로 정리해 줘. 가정·한계가 있으면 한 문장으로 명시해 줘. |
| **C** | **2-Tier 개념**을 답변에 끌어오기 | 이 시스템에서 수치·예측·집계는 어디서 하고, 요약·해석은 어디서 하는지 이 데이터셋 예시에 맞춰 한국어로 짧게 설명해 줘. |
| **D** | SCM 성격 강조 | 수요·재고 관점에서 이 데이터셋이 시사하는 점과, 의사결정 시 확인해야 할 점을 bullet로 한국어로 정리해 줘. |
| **E** | CRM 성격 강조 | 이탈·고객 가치 관점에서 우선 행동을 어떻게 가져가면 좋을지, 한국어로 실무자에게 전달할 만한 형태로 제안해 줘. |
| **F** | BI 성격 강조 | 지역·기간·성장률 같은 트렌드 관점에서 이 데이터셋의 메시지를 한국어로 요약해 줘. |
| **G** | 후속 액션(교육용) | 위 분석을 바탕으로 다음 주 액션 아이템을 우선순위대로 최대 5개, 한국어 bullet로 제안해 줘. |
| **H** | (선택) 영어 | Summarize the main SCM, CRM, and BI insights for this dataset in English in one short paragraph. |

**시연에서 고르는 법**

- **첫 시연·시간 여유**: **A** 또는 **B**.  
- **앞에서 SCM 탭만 돌린 직후**: **D**만 써도 **Tier2가 무엇을 하는지** 보여 주기에 충분하다.  
- **2-Tier 슬라이드와 연결**: **C**를 쓰면 **다이어그램·슬라이드**와 멘트가 맞물린다.  
- **마무리 한 방**: **A′** 또는 **G**.

#### 6-4) 시간이 촉박할 때

- **Tier2 스킵**: 다이어그램에서 Tier2만 짚고, 질의 실행은 생략한다.  
- **30초만**: **A′** 한 번만 실행하고 textarea 결과 **2~3줄만** 소리 내어 읽는다.

### Dify를 따로 둔 이유 vs 로컬 **Ollama 3.2** (교육생이 “왜 둘 다?”라고 할 때)

| | **Dify** (self-hosted) | **Ollama 3.2** (로컬 LLM) |
|---|------------------------|---------------------------|
| **한 줄** | **AI 오케스트레이션** — 워크플로 캔버스, HTTP Request로 FastAPI 호출, 변수·조건·LLM 노드 체인, 앱/에이전트 실행 UI | **추론 엔진** — 설정된 모델로 채팅/완성 **토큰 생성**만 담당 |
| **이 프로젝트에서** | `demo/DEMO_SCRIPT`의 Studio 시연, 백엔드 **`/agent/query`** 가 내부적으로 타는 **앱·워크플로** 런타임, `compact=true`로 받은 JSON을 LLM에 넘기는 **파이프라인** | 강의 PC에 깔아 **비용·데이터 유출 최소화**로 Tier2에 붙인 **실제 LLM 백엔드** (클라우드 API 대체 가능) |
| **비유** (`dify_integration.md`) | Camunda 같은 **BPM + LLM 노드**에 가깝다 | 엔진 룸의 **모델 서버**에 가깝다 |

**강조 멘트 예**: “**Ollama만** 있으면 터미널·간단 API로 채팅은 되지만, **FastAPI → 여러 HTTP → 합쳐서 요약** 같은 **교육용·운영용 파이프라인**을 **화면에서 조립·게시**하려면 **Dify**가 필요하다. 반대로 **Dify만** 있고 로컬 모델이 없으면, 우리가 쓰는 것처럼 **Ollama를 ‘LLM 제공자’로 등록**해 연결한다.”

**하지 말 것**: “Dify가 숫자를 계산한다” — **금지**. 숫자는 **항상 Tier1**.

### 슬라이드 6 — 데모 동선 + 스토리라인 (브라우저 시연 구간)

> **덱 6.** 불릿: 접속 → 업로드·`dataset_id` → SCM/CRM/BI → Dify 요약/해석. **일일 대본**: **`A01`** 도입 뒤 이어 붙이거나, **`A03` 브리지 직전**에 실행. **화면 세부**는 **「1)~5)」**, **스토리라인**은 **「스토리라인·전제」**, **흐름 그림·Dify 탭 질의**는 **「6)」**. **다음 슬라이드 7**은 **`A02`**(2-Tier·스택).

“(슬라이드 불릿 짚기) 오후에는 **한 장으로 동선**만 고정합니다. **데모 접속** → **CSV 업로드로 `dataset_id`** → **SCM·CRM·BI 탭**에서 Tier1 분석 → **Dify 쪽**으로 요약·해석을 넘깁니다.

(스토리라인 — **20~60초**, 시간 있으면) 이 화면과 API는 **제가 해당 도메인을 깊게 알고 시작한 프로젝트는 아니고**, **AI로 코드베이스와 흐름을 분석·구축하는 데 도움을 받아** 대략 **며칠(3~4일) 안에** **끝까지 한 줄기**를 올린 **1차 버전**입니다. 그래서 **납품 완성도**라기보다 **계약·파이프라인 시연**에 가깝고, **맞는 다음 단계**는 **현장 도메인으로 핵심 규칙과 지표를 다시 정의**한 뒤 **그에 맞게 솔루션을 보강**하는 쪽이라고 보시면 됩니다. 오전에 말씀드린 **문서·Gate**는 그 **짧은 기간 안에도** 통제 축을 같이 깔기 위한 것이었습니다.

(브라우저로 전환) **`demo/index.html`** 맨 위 **`#demo-flow-overview`** 다이어그램을 **30초~1분** 짚고(위 **「6-1」** 순서), 이어서 업로드·탭 시연. **「Dify AI 요약」** 탭에서는 **「6-3」** 표에서 **질의 문장 하나**를 복사해 **에이전트 실행**한다.” *(선행 조건은 **「6-2」**·세부는 **1)~6)** 참고.)*

(슬라이드로 복귀) **다음 슬라이드 7**에서는 아까 그림과 **같은 2-Tier 축**을 **스택 이름**(FastAPI·Pandas·PostgreSQL·Redis/ARQ·Chart.js·Dify·LLM 등)으로만 다시 고정합니다 — 일일 대본 **`A02`**.”

### 슬라이드 7 — 2-Tier 재확인 + 기술 스택 (`A02`)

> **덱 7 · 트랙 A02.** 아까 브라우저 **그림과 같은 축** — 여기서는 **제품명·스택**만 실명으로 고정. 상세는 위 **「실제 작성 덱 — 슬라이드 7」**.

“(슬라이드) 방금 데모 **맨 위**에서 **화살표로 구조**는 보셨습니다. 이 슬라이드는 **그림을 다시 그리지 않고**, **같은 2-Tier**를 **실제 스택 이름**으로만 재확인합니다.

**Tier1**은 **FastAPI**와 **Pandas**, 데이터는 **PostgreSQL**, 오래 걸리는 잡은 **Redis**와 **ARQ** 워커 — 방금 보신 **차트**는 브라우저 **Chart.js**가 Tier1이 준 **JSON**을 그린 겁니다. **Prophet**이나 **RFM·K-Means** 같은 **알고리즘은 전부 이 계층**입니다.

**Tier2**는 **Dify**가 **워크플로·HTTP 체인**을 짜고, **말을 실제로 생성하는 LLM**은 로컬 **Ollama 3.2**에 붙였습니다. 브라우저는 **`/agent/query`** 로만 치고, **백엔드가 Dify를 거쳐 Ollama**까지 연결합니다. **Dify는 수치를 계산하지 않고**, **Ollama는 파이프라인을 구성하지 않습니다.**

**complexity**는 한 줄로만 — **단순 분석**은 Tier1만, **복합·자연어**는 Tier2로 넘기고, LLM 쪽은 **`compact=true`** 같은 짧은 JSON 계약을 쓰는 팀 규칙이 붙어 있습니다.

다시 **브라우저**로 돌아가면, 일일 대본 **`A03`** 한 줄 짚고 **SCM·CRM·BI 탭**을 순서대로 보겠습니다.”

### 슬라이드 11 — 마무리: 보강 방향 (`R01`)

> **덱 11 · 트랙 `R01`** · 세그먼트 **`WRAPUP`** · 출처 **call + plan**  
> 오전·오후 **스토리라인**(도메인 비전문·1차 축)과 **맞물리게** 마지막 슬라이드를 잡는다. **(구)** 안건이 “성과를 KPI와 연결 + 신규 기능 3축 + 프로토타입” 중심이었다면, **이 저장소 시연 서사**에는 **위「실제 작성 덱 — 슬라이드 11」** 의 **권장 불릿**이 더 잘 맞는다.

“(슬라이드) 오늘 데모는 **짧은 기간에 올린 1차 축**이었고, **납품 완성도**보다 **계약·파이프라인**을 보여 드리는 데 목적이 있었습니다. 마지막으로 **앞으로 보강할 방향**만 세 줄로 고정하겠습니다.

**첫째**, **핵심 도메인**에서 용어·KPI·데이터 품질을 원천으로 **규칙과 스키마를 다시 잡는 것**입니다. 어떤 열이 필수인지, `dataset_type`·`period`·검사·고객 코드가 무엇을 의미하는지, **`compact` 응답**에 무엇을 남길지, Dify 프롬프트와 검증 시나리오까지 **같은 도메인 계약**으로 묶습니다.

**둘째**, 그 계약 위에서 **RAG**로 내부 문서·로그·SOP를 붙이고, **에이전트**로 Tier1 API를 **도구 호출**로 묶는 쪽으로 확장할 **준비**를 합니다. 관계·메타데이터가 중요해지면 **GraphRAG** 같은 **구조화 지식** 쪽도 검토합니다 — 단, **수치는 여전히 Tier1**이라는 전제는 유지합니다.

**셋째**, **AI 백엔드와 웹 프로토타입**은 그대로 두되, 우선순위는 **도메인 규칙 → 스키마·API → 그다음 LLM 확장** 순으로 잡는 것이 안전하다고 보시면 됩니다.

(질문 받기) KPI 대시보드나 3축 기능 로드맵은 **조직 과제**로 남기고, **기술 스택 관점**에서는 오늘 말씀드린 **도메인 계약 + RAG/에이전트 준비**를 기억해 주시면 됩니다.”

## 참고 — 지시→통제 전환 (심화·Q&A용)

강의 **본편에 넣지 말고**, “왜 프롬프트만으로 안 되고 문서·금지로 갔나?” 질문이 길게 나올 때만 사용한다. 원문·날짜별 기록은 `docs/rules/error_analysis.md`.

**프롬프트만으로 한계였던 이유 (요지)**

- 세션·팀원마다 지시문이 달라져 **재현성**이 떨어짐. 모델은 **한 턴에 끝내기** 쉬워 승인·테스트가 빠지기 쉬움. 채팅은 **계약·감사**로 남기 어렵다. 금지·게이트·필독을 **레이어와 도구**로 쪼개 두어야 같은 문장이 반복 주입된다.

**실제로 반복된 패턴 (`error_analysis`와 대응)**

- 구현 직후 확인·테스트 게이트 생략 (2026-03-26) → Gate B 멈춤·승인 전 금지 명시.  
- 작업 계획을 `CURRENT`가 아닌 별도 파일로만 둠 (2026-03-26) → 상세 계획 위치 고정.  
- 부분 스테이징 + pre-commit stash로 같은 린트 반복 (2026-03-26) → 커밋 전 순서·전부 스테이징 문서화.  
- MCP·문서 해석을 배포까지 확대 (2026-03-27~28) → 엣지 nginx 범위를 스킬·컨텍스트에 표로 고정.  
- 공인 URL 문서에서 비표준 조치를 먼저 제안 (2026-03-28 전후) → `lis_public_url_path_map.md` 앞부분에 운영 원칙 한 줄 고정.

**멘트 한 줄**

“**잘하라**는 말은 매번 새로 써야 하고, **하면 안 되는 일**은 저장소에 박아 두고, 같은 사고는 `error_analysis`에 쌓아 문장으로 다시 흡수하는 **루프**로 가져갑니다.”
