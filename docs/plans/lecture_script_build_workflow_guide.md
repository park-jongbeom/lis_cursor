# 강의 자료 제작 워크플로 가이드 (슬라이드 먼저, 대본 나중)

> 목적: 먼저 Apps Script로 PPT 골격을 빠르게 만든 뒤, 현재 프로젝트 정보를 근거로 강의용 대본을 별도 구성한다.
>
> 대상 문서:
> - 슬라이드 자동 생성: `docs/plans/ppt_google_slides_autogen_script.md`
> - 슬라이드 내용 상세: `docs/plans/ppt_aux_instructor_build_guide.md` (상단 **문서 보강 작업 기록** = PPT 가이드 보강·이어하기 마스터 로그)
> - 프로젝트 근거: `docs/plans/plan.md`, `docs/CURRENT_WORK_SESSION.md`, `docs/rules/*.md`, `demo/DEMO_SCRIPT.md`, `demo/README.md`

---

## 1) 전체 전략

- **1단계(빠른 골격)**: Apps Script로 슬라이드 본문을 일괄 생성
- **2단계(강의 대본화)**: 생성된 슬라이드 순서대로 “발표 대본”을 별도 문서로 작성
- **3단계(리허설 최적화)**: 시간대별(오전/오후)로 대본 길이와 데모 동선을 압축

핵심 원칙:

- 슬라이드와 대본을 한 번에 완성하려고 하지 않는다.
- 먼저 “흐름”을 고정한 뒤, 대본은 청중 수준(실무 Java 백엔드) 기준으로 정밀화한다.

---

## 2) 단계 1 — 슬라이드 골격 먼저 만들기

### 2-1. 생성 규칙

- 디자인은 나중에 처리한다.
- 텍스트(제목/불릿/노트)와 이미지 위치만 먼저 만든다.
- 자동 생성은 `createDeck()` 1회 실행으로 끝낸다.

### 2-2. 실행 절차

1. Google Slides에서 빈 프레젠테이션 생성
2. Apps Script 열기
3. `docs/plans/ppt_google_slides_autogen_script.md`의 코드 붙여넣기
4. `createDeck()` 실행
5. 생성 결과 확인(슬라이드 수/순서/노트)

### 2-3. 결과 확인 체크

- AI 도입(A*) → 2회차 준비(E*) → 샘플 본론(S*) → AI 정리(R*) 순서인지
- 이미지 삽입 안내 텍스트박스가 필요한 슬라이드에 들어갔는지
- 발표 노트가 각 슬라이드에 들어갔는지

---

## 3) 단계 2 — 강의 대본 문서 별도 구성

> 이 단계부터 “강의용 대본 파일”을 따로 만든다.

권장 파일:

- `docs/plans/lecture_talk_track_2026_03_28.md`

### 3-1. 대본 문서 구조(권장 템플릿)

각 슬라이드마다 아래 6개 항목을 고정한다:

1. 슬라이드 ID/제목
2. 20초 요약 멘트
3. 60초 상세 멘트
4. 기술 심화 포인트(Java 백엔드 실무자용)
5. 데모 전환 멘트(다음 슬라이드 연결)
6. 실패 시 대체 멘트(플랜B)

### 3-2. 청중 수준 반영 규칙

- “AI 사용법 소개”보다 “개발 공정 통제”를 강조
- 용어는 실무 프레임으로 맞춘다:
  - 계약(Contract), 상태 머신(State Machine), 증적(Evidence), 가드레일(Guardrail), 운영 표준
- 예시는 Spring/Spring Boot 관점 비유로 전달
- 모델 비교는 감성 추천이 아니라 “작업 유형/비용/품질” 3축으로 설명

---

## 4) 단계 3 — 현재 프로젝트 정보를 대본에 주입하는 방법

## 4-1. 반드시 반영할 근거 포인트

- `docs/plans/plan.md`
  - 프로젝트 Phase 진행 상태
  - 강의 데모 목표(SCM/CRM/BI)
- `docs/CURRENT_WORK_SESSION.md`
  - 현재 세션 상태/Gate 맥락
- `docs/rules/workflow_gates.md`
  - A~E 승인 경계
- `docs/rules/error_analysis.md`
  - 실제 실패 유형과 재발 방지 사례
- `demo/DEMO_SCRIPT.md`, `demo/README.md`
  - 당일 시연 순서와 우회 시나리오

### 4-2. 슬라이드 그룹별 메시지

- **A(도입)**: “왜 AI 공정 통제가 필요한가”
- **E(2회차 준비)**: “도입 준비/개발환경/모델 운영”
- **S(본론)**: “실제 데이터분석/아키텍처/코드 연결”
- **R(정리)**: “실패 학습/품질/보안/복제 템플릿”

---

## 5) 강의 당일 운영 가이드(실전)

### 5-1. 시간 배분

- 1부(A): 5~8분
- 1.5부(E): 5~8분
- 2부(S): 20~30분
- 3부(R): 5~7분

### 5-2. 데모 중심 운영 원칙

- 설명이 길어지면 즉시 화면으로 전환
- 모델/툴 논쟁보다 “팀 표준화 포인트”로 마무리
- 장애 발생 시 원인 분석보다 우회 루틴 우선

### 5-3. 현장 질문 대응 프레임

- “이 모델이 최고인가요?” → 작업 유형 기준 라우팅 설명
- “비용이 부담되는데?” → 고급 모델은 상위 단계 집중, 하위 반복 작업은 가성비 모델
- “팀 적용은 어떻게?” → 규칙 문서 + Gate + 품질 게이트 + 포스트모템 루프 순서로 답변
