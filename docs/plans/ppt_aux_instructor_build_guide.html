  4. 통제가 목표(슬라이드 3번 불릿): 속도가 아니라 통제 가능한 공정. 질문이 “왜 규칙이 많나?”로 가면 짧게: 긴 프롬프트만으로는 같은 실수가 반복되기 쉬워, 금지·게이트·error_analysis 누적으로 옮겼다 — 필요 시 목차 #c-shift 또는 #c-stories에서 카드 1장만 보여 주고, 상세는 본 문서 말미 「참고 — 지시→통제 전환」 을 발표자가 읽는다.

덱 번호 · 트랙 (정본)
- 슬라이드 4 = 본 절(Gate A~E).
- 일일 대본 M04 후반에서 Gate 멘트를 연다(같은 M04 안에서 슬라이드 3 규칙 체계에 이어짐).

슬라이드 초안 검토 (정본: docs/rules/workflow_gates.md)

초안:

  • 제목: Gate A~E로 승인 경계 분리
  • A: 구현 계획 승인
  • B: 구현 완료 후 강제 정지
  • C/D/E: 테스트 계획 / 실행 / 이력 전환
항목 판단 권장
전체 구조 맞다. 구현 권한과 테스트 권한을 승인 단계로 쪼개는 요지가 정본 다이어그램과 같다. 유지.
A “구현 계획”만 있으면 코딩 전인지·어디에 쓰는지가 빠지기 쉽다. 정본은 Gate A = CURRENT에 구현 상세 계획을 쓰고 사용자 승인 전 코드 변경 금지다. 슬라이드에 「코딩 전」 또는 「CURRENT에 상세 계획」 중 하나를 넣는 것이 좋다. (예: A: 코딩 전 — 구현 상세 계획 승인.)
B “강제 정지”는 팀 용어로 Gate B 직후 멈춤(구현 끝 → 사용자 확인 → 그다음 C)과 잘 맞는다. 유지. 발표에서 테스트로 바로 가지 않는다를 한 번 더 말한다.
C 정본에서 C는 테스트 실행 전 단계다. “테스트 계획”만 있으면 승인 후에만 D(실행) 라는 점이 약해진다. C: 테스트 상세 계획 승인(실행 전) 정도로 쓰면 AI 과대해석 방지 메시지와 맞는다.
D “실행”에 결과 기록이 포함된다는 점을 슬라이드에 쓸 공간이 있으면 좋다. 예: D: 테스트 실행·검증 결과 기록.
E “이력 전환”은 맞다. 정본은 WORK_HISTORY + plan.md 갱신 + 다음 세션용 CURRENT로 교체까지 포함한다. 필요 시 괄호로 「다음 세션 문서」 한 단어 추가.

한 줄 정본 출처 (발표자·슬라이드 각주용): workflow_gates.md 요약 다이어그램 — A·B·C 각각 사용자 승인(⏸) 이 있고, 「Gate A 승인 = 구현 착수만」 이 테스트까지 포함하지 않는다는 점이 중요하다.

왜 슬라이드 3에서 Gate를 짚은 뒤, 슬라이드 4에서 다시 설명하는가

슬라이드 3(규칙 웹·브라우저)에서는 의도적으로 Gate를 표 한 장·흐름 몇 마디로만 끝낸다. 같은 날 오전 흐름에서 Gate를 두 번 다루는 것은 중복이 아니라 역할 분담이다.

  1. 인지 부담 분리
    슬라이드 3은 레이어(L1/L4/L5)·세션 문서·ZIP·통제 목표까지 한꺼번에 올라간다. 여기서 A~E를 글자 하나하나까지 풀면 청중은 “규칙 시스템 전체”와 “승인 상태 머신”이 한 덩어리로 뭉개져 기억하기 어렵다. 그래서 3에서는 「금지 열이 있고, 계획→구현→확인→테스트가 끊긴다」 수준만 남기고 끝낸다.

  2. 두 번째 슬라이드 = 추상 → 운영 규칙
    3번은 지도(어디에 뭐가 있는지) 이고, 4번은 운전 매뉴얼(어떤 순서로 손대면 안 되는지) 에 가깝다. 특히 B에서 멈춤, A 승인이 테스트까지가 아님은 AI·사람 모두 자주 깨는 지점이라(error_analysis, workflow_gates.md 「승인 범위 해석」), 슬라이드 한 장을 통째로 할애하는 것이 교육 효과 대비 비용이 낮다.

  3. 오전 스크립트(M04 → M05)와의 정렬
    M04: 규칙 체계(슬라이드 3)에 이어 Gate(슬라이드 4) 멘트. M05: 슬라이드 5 Swagger·OpenAPI. 덱에서 3번에 Gate 표를 잠깐 보여 준 것은 4번 승인 경계의 예고로 쓴다.

  4. 현장 질문 대비
    “진행해라 한 마디로 테스트까지 돌린 적 있다”는 식의 질문은 3번 맥락만으로는 답이 길어진다. 4번에서 A/B/C에 각각 승인이 있다는 것을 한 번 더 박아 두면, 오후 실습·코드 시연으로 넘어갈 때 프로세스 논쟁이 줄어든다.

  5. 반복의 법칙
    교육에서 핵심 안전 규칙은 “한 번에 크게 + 나중에 같은 주제만 좁혀서” 두 번 말하는 편이 유지된다. Gate는 품질·권한 사고와 직결되므로 의도적 반복으로 본다.

정리 (발표자 한 줄): “아까 웹에서는 Gate가 전체 그림 속 한 조각이었고, 지금 슬라이드는 승인만 잘라서 보여 드립니다.”

슬라이드 4 — 전달 순서·시간

  • 1분(압축): 제목 + 불릿 읽기 + “A 승인은 구현만, B 끝나면 멈춤, 테스트는 C 이후” 한 문장.
  • 2~3분(권장): 위 + 승인 과대해석 금지(정본 한 줄: A 승인 ≠ C/D 자동 승인) + 질문 1회 받기.
  • 시각 자료: 가능하면 workflow_gates.md의 텍스트 다이어그램을 슬라이드에 옮기거나, 규칙 웹의 Gate 표(#c-gates)를 슬라이드 4에서만 다시 띄워 같은 표를 두 번째 관점으로 읽는다(“이번에는 열보다 멈춤 지점만”).

캡처·자료

목적에 대한 판단

“AI로 프로젝트를 구축하면서 Swagger까지 자동으로 나온 것을 보여준다”는 방향은 맞다. 이 저장소는 FastAPI라서, 엔드포인트·Pydantic 스키마·태그 등이 잡히면 /docs·/openapi.json이 코드에서 파생된다. 즉 별도 스펙 문서를 사람이 매번 맞춰 쓰지 않아도 계약 표면이 생긴다는 점을 오전 마무리로 쓰기 좋다.

정확한 한 줄: “AI가 OpenAPI YAML을 손으로 다 쓴 것”이 아니라, AI·사람이 쓴 라우트/모델 위에 FastAPI가 스펙을 생성한다. 그래도 청중 입장에서는 “코드 기반으로 스웨거가 따라왔다”는 인상 목적은 달성된다.

Swagger UI에 보이는 “설명”은 어디서 오나
화면에 읽히는 요약·설명·파라미터 설명은 프레임워크가 공중에서 만드는 게 아니라, 코드 안 메타데이터에서 온다. 예: 엔드포인트 summary / docstring, Pydantic Field(description=...), 라우터 tags, response_model 등. 이걸 비워 두면 경로·타입만 나오고 칸이 허전해진다.

이 프로젝트 서술 보강(권장)
위 메타데이터는 사람이 전부 손으로만 채운 것도, 전부 자동도 아니다. 실제 흐름은 AI가 라우트·스키마와 함께 설명 문구 초안을 같이 쓰고, 규칙·리뷰·Gate·린트로 다듬은 뒤 코드에 남기므로 스웨거에도 그대로 반영된다고 말하면, “FastAPI가 알아서 다 해준다” 오해와 “설명은 따로 문서 팀이 써야 한다” 오해를 동시에 줄일 수 있다.

트랙 ID: 본 슬라이드는 일일 대본 M05 와 덱 슬라이드 5에 고정한다(이전에 문서에 적혀 있던 M07 표기는 폐기).

1) 사실 관계(강조해도 되는 문장)

  • 본인은 이 데모가 다루는 세부 도메인(예: 해당 업종 KPI·현장 의사결정 규칙) 에 대해 전문가 수준의 지식을 갖고 시작하지 않았다.
  • 코드베이스·API·샘플 데이터·화면 흐름은 AI 도구를 활용해 시스템을 분석·설계하고, 짧은 기간(총 약 3~4일 정도의 오더) 안에 수직 슬라이스(업로드 → Tier1 분석 → Tier2 요약 연결) 를 동작 가능한 형태로 올린 사례다.
  • 따라서 오늘 보여 주는 것은 “현장에 그대로 납품할 최종 솔루션”이라기보다, 교육·시연·계약(API)·통제 축(Gate·규칙 문서)이 같이 깔린 1차 버전에 가깝다.

2) 왜 솔직한 전제가 유리한가

  • 청중(실무 개발자)은 “AI로 빠르게 만들었다”는 말만 들으면 품질·책임을 의심하기 쉽다. 도메인을 깊이 모른 채 시작했다고 말하면 오히려 과장이 아닌 투명성으로 읽힌다.
  • 동시에 오전에 이미 설명한 .cursorrules·docs/rules·Gate·테스트와 연결하면, “빠른 구축 = 아무 규칙 없이”가 아니라 “짧은 시간 안에도 계약·승인·증적 축을 같이 깔았다”는 메시지가 된다.
  • 속도만 강조하지 말고 반드시 한 짝을 붙인다: “그만큼 다음 레이어에서 실제 도메인으로 규칙과 지표를 다시 잡고 솔루션을 보강할 예정이다.”

3) “3~4일”이 의미하는 것·의미하지 않는 것

의미해도 되는 해석 피해야 하는 오해
교육용 수직 슬라이스를 API·UI·LLM 연동까지 한 흐름으로 만든 캘린더 기준 기간 “업계 표준 컨설팅 산출물 전부 완료”
샘플 CSV·기본 컬럼 가정 위에서 파이프라인·계약을 증명 “특정 고객사 실데이터·법규·승인 프로세스까지 반영 완료”
AI가 초안·반복 작업을 많이 맡고, 사람이 방향·승인·문서·린트로 다듬은 기간 “버튼 한 번으로 전부 자동 완성”

4) AI가 한 일 vs 사람·저장소가 한 일 (발표에서 짧게 나누는 법)

  • AI 쪽: 요구를 질문으로 쪼개기, 라우트·스키마·데모 HTML·Dify 연동 초안, 문서 초안, 반복 수정.
  • 사람·저장소 쪽: plan/CURRENT·Gate로 승인 경계, docs/rules로 아키텍처 계약, ruff·mypy·pre-commit·테스트로 증적, error_analysis로 실패 모드 축적.
  • 한 줄 요약 예: “뼈대와 초안의 속도는 AI가, 무엇을 하면 안 되는지와 언제 멈출지는 저장소가 잡는다.”

5) 다음 단계(로드맵) — 강사가 말하기 좋은 ‘올바른 성숙 방향’

  • 실제 도메인에서: 용어 정의, KPI·임계값, 데이터 품질 규칙, “이 지표로 무슨 결정을 하는가”를 원천으로 삼는다.
  • 그에 맞춰 핵심 규칙을 재정립한다: 어떤 열이 필수인지, compact에 무엇을 넣을지, 어떤 분석은 Tier1에 남기고 어떤 해석은 Tier2에 둘지, Dify 프롬프트·검증 시나리오.
  • 그다음 솔루션을 보강한다: 스키마·엔드포인트·잡 파라미터·샘플이 아닌 운영 데이터 경로, 모니터링, 접근 통제.
  • 이 순서를 말하면 “AI로 때운 데모”가 아니라 “1차 계약을 깐 뒤 도메인으로 정식화한다”는 엔지니어링 서사가 된다.

6) 슬라이드에 넣을 수 있는 선택 불릿 (슬라이드 6에 한 줄 추가할 때)

  • (선택) “도메인 전문가로 시작하지 않았고, AI 보조로 약 3~4일 안에 동작하는 데모 축을 올린 사례”
  • (선택) “다음은 현장 도메인으로 규칙·지표를 재정의하고 그에 맞게 보강하는 단계”

7) 발표 시간별 멘트 예시 (발표자 노트에 그대로 써도 됨)

  • 20초: “이 데모는 해당 도메인을 깊게 알고 시작한 프로젝트는 아니고, AI로 시스템을 분석·구축하는 데 도움을 받아 며칠 안에 화면·API·요약까지 연결한 1차 버전입니다. 다음은 실제 도메인으로 규칙을 다시 잡고 솔루션을 보강하는 방향이 맞다고 봅니다.”
  • 60초: 위 20초 문장에 이어서: “3~4일은 교육용으로 끝까지 한 줄기를 만든 기간이지, 현장 납품 전 과정이 아닙니다. 빠른 만큼 오전에 말씀드린 문서·Gate·테스트로 통제 축을 같이 깔았고, 도메인이 들어오면 지표 정의·스키마·Dify 쪽을 그 규칙에 맞춰 고치는 게 정석입니다.”
  • 2분(오후 도입 + 슬라이드 6 불릿 짚기 직전): 60초 버전에 위 4) AI vs 저장소 요지를 한 번 더 짚은 뒤, “지금부터 보실 화면은 그렇게 올린 저장소의 실제 demo/index.html입니다”라고 하고 아래 UI 절「1)~5)」 로 넘어간다.

8) 하지 말 것

  • “전문가급 도메인 솔루션이다”, “현장에 바로 쓰면 된다”처럼 검증되지 않은 납품 수준으로 끌어올리지 않는다.
  • 3~4일만 내세우고 규칙·검증을 빼먹지 않는다 — 오전 메시지와 모순이 된다.

3) SCM / CRM / BI 탭 — 화면에 실제로 있는 버튼과 산출물

탭 바는 네 칸이다. 이름은 데모 HTML과 글자까지 맞출 것: 「SCM 수요예측」, 「CRM 이탈·클러스터」, 「BI 트렌드」, 「Dify AI 요약」(마지막 칸은 다음 소절).

SCM 수요예측 (#panel-scm)

  • 입력: target_column, date_column, group_by, test_codes, forecast_days — 기본값은 scm_sample.csv 에 맞춰져 있다.
  • 예측 실행 → 폴링: POST /scm/forecast 후 /scm/forecast/{job_id} 를 최대 약 180초 폴링. 아래 라인 차트(Chart.js)에 상위 시리즈 예측이 그려진다.
  • 상태 줄(#statusScm): 잡 ID, 완료 시 모델명·시리즈 개수 등.
  • 주의: ARQ 워커가 없으면 잡이 pending 에서 멈출 수 있다 — demo/DEMO_SCRIPT.md 리허설 체크리스트와 동일.

CRM 이탈·클러스터 (#panel-crm)

  • 이탈 위험 분석: 동기 GET /crm/churn-risk — 바 차트 + 코드/이름/경과일/위험/세그먼트 테이블.

  • 두 버튼 모두 같은 캔버스·표 영역을 쓰므로, 시연 순서를 정해 두면 청중이 덜 헷갈린다(예: 이탈만 또는 클러스터만).

BI 트렌드 (#panel-bi)

  • 지역 히트맵: GET /bi/regional-heatmap — period는 CSV의 기간 값과 문자열이 정확히 같아야 한다(샘플 기본 2024-01). 표 + 막대 차트, 인사이트 문장.
  • YoY 비교: GET /bi/yoy-comparison — 연도별 YoY 막대 차트.
  • 둘 다 동기 호출이라 워커 없이도 동작하기 쉬워 시연 폴백으로 좋다.

  • 의미: 먼저 CRM compact JSON을 받고, 이어서 BI compact JSON을 받은 뒤, 집계 노드가 두 HTTP 응답 body를 합쳐 문자열로 만든다. 그 문자열과 시작 노드의 user_query 를 LLM에 넣어 최종 텍스트를 낸다.

1) 시작 노드 (start)

변수 타입 필수 설명
user_query paragraph 예 최종 LLM에 같이 넘어가는 자연어 질문 (에이전트·앱 입력과 맞출 수 있음).
dataset_id text 예 Tier1 GET /crm/churn-risk, GET /bi/regional-heatmap 에 공통으로 쓰는 UUID. 데모 HTML의 녹색 dataset_id 와 동일 값을 복사해 넣는다.
period text 예 BI 히트맵 쿼리용. CSV period 열 값과 문자열이 정확히 같아야 한다 (예: 샘플 bi_sample.csv 계열이면 2024-01). 노드 설명에도 “regional-heatmap 필수”라고 적혀 있는 맥락과 동일.

시연 멘트 예: “시작 노드에 API가 요구하는 키만 박아 두었습니다. dataset_id는 아까 업로드한 것과 같고, period는 BI 샘플과 글자 하나까지 맞춰야 합니다.”

2) HTTP 노드 ① — CRM churn-risk

  • 메서드: GET
  • 경로 개념: /api/v1/crm/churn-risk
  • 쿼리 (DSL 패턴): dataset_id={{시작.dataset_id}}, compact=true, top_n=10 (또는 DSL에 맞춘 값).
  • 인증: 헤더 Authorization: Bearer <JWT> — Dify UI에서만 설정. 문서·채팅·DSL Git 커밋에 실토큰을 넣지 말 것.
  • 역할: 이탈 위험 상위 N 등 CRM 요약 JSON을 Tier1에서 가져온다 — 계산은 전부 FastAPI.

3) HTTP 노드 ② — BI regional-heatmap

  • 메서드: GET
  • 경로 개념: /api/v1/bi/regional-heatmap
  • 쿼리 (DSL 패턴): dataset_id={{시작.dataset_id}}, period={{시작.period}}, compact=true.
  • 인증: 동일하게 Bearer JWT.
  • 역할: 지역·기간 관점 BI 요약 JSON을 가져온다. period 불일치 시 404/에러가 나기 쉬움 — 리허설 필수.

4) 변수 집계자 (variable-aggregator)

  • 입력: 위 두 HTTP 노드의 body (응답 본문).
  • 출력: 하나의 문자열(output)로 이어 붙인 뒤 LLM 노드로 전달.
  • 시연 멘트 예: “여기서 두 번의 Tier1 호출 결과를 한 덩어리 텍스트로 합칩니다. LLM은 원시 CSV를 안 보고, 이미 압축된 JSON 문자열만 봅니다 — 그래서 compact=true가 규칙으로 박혀 있습니다.”

5) LLM 노드 — LLM (Ollama)

  • 제공자 (DSL): langgenius/ollama/ollama 플러그인.
  • 모델명 (DSL 예시): alibayram/Qwen3-30B-A3B-Instruct-2507:latest — 강의 PC Ollama에 실제로 pull·등록된 이름과 맞춰야 한다. (다른 모델을 쓰면 Dify에서 모델 선택만 바꾸면 됨.)
  • 옵션 (DSL): temperature: 0.3, 모드 chat.
  • 시스템 프롬프트 (DSL 요지): “당신은 의료 검사 영업 분석 전문가입니다. 주어진 JSON을 바탕으로 한국어로 실무적 인사이트를 제시하세요. 수치를 지어내지 말고, 과장한 해석은 피하세요.”
  • 유저 프롬프트 구조 (DSL 요지)
  • 데이터 (연결된 compact JSON): + {{변수 집계자.output}}
  • 사용자 질문: + {{시작.user_query}}
  • “위 데이터와 질문에 대해 한국어로 답변하세요.”

시연 멘트 예: “숫자는 JSON에만 있고, 모델은 그 JSON과 질문만 받습니다. 시스템 역할은 도메인 톤(여기서는 의료 검사 영업)을 잡아 준 것이고, 강의 목적이면 나중에 일반 SCM/CRM 문구로 바꿀 수 있습니다.”

6) 끝 노드 (end)

  • 출력 변수: LLM 노드의 text → 앱 실행 화면에 최종 답변으로 표시.

리허설 체크리스트 (Studio 진입 시)

  1. FastAPI·공인/로컬 URL이 Dify HTTP 노드와 동일 베이스인지 확인.
  2. JWT 갱신 후 두 HTTP 노드 헤더에 반영 (DEMO_SCRIPT.md·make dify-fastapi-jwt-bearer).
  3. 데모에서 dataset_id 복사 → Dify 실행 입력에 붙여넣기.
  4. period = 해당 데이터셋 CSV의 기간 값과 완전 일치.
  5. 한 dataset_id로 CRM·BI API가 둘 다 유의미한 JSON을 주는지 미리 확인 — crm 전용·bi 전용 CSV만 올렸다면 한쪽 HTTP는 에러·빈 응답이 될 수 있음. 강의용으로는 두 엔드포인트가 모두 성공하는 업로드 조합을 리허설에서 고정하는 것이 안전하다.
  6. Ollama 기동·모델 태그가 LLM 노드 설정과 일치.
  7. Publish 된 워크플로/앱인지 확인 후 시연.

슬라이드·말하기 연결

  • 슬라이드 7에서 말한 “Dify = 오케스트레이션, Ollama = 추론” 을 이 캔버스에 대응시키면: HTTP·집계 = Dify가 짠 파이프, LLM 블록 = Ollama에 붙은 모델.
  • 데모 HTML agent/query 와 같은 Tier1 계약을 쓰되, Studio 버전은 노드가 보이므로 “교육용 투명성”이 좋다고 한 줄 말하면 된다.

5) 슬라이드에 넣기 좋은 캡처·리허설 체크

  • 한 장 요약용 스크린샷: 가능하면 첫 패널 + 탭 네 칸이 한 화면에 들어가는 폭으로 찍거나, 상단 패널과 탭 바만 크롭.
  • 리허설 (demo/DEMO_SCRIPT.md와 동일 계열): health·로그인(또는 자동 로그인)·샘플 업로드 한 번·SCM 또는 BI 한 경로 성공·가능하면 에이전트 탭 1회.
  • 시간이 촉박할 때: 샘플 즉시 업로드 → CRM 이탈(동기) 또는 BI 히트맵 → (시간 되면) 에이전트 한 줄.

데모 HTML 상단 2-Tier·complexity 그림은 슬라이드 없이 구조를 고정할 때 쓴다(6-1).
여기서 말하는 질의는 교육생이 강사에게 하는 질문이 아니라, 강사가 데모 화면 「Dify AI 요약」 탭의 질의 입력란(#agentQuery)에 붙여 넣어 에이전트 실행 할 자연어 프롬프트 목록이다. 발표자 노트·리허설에 복사해 쓴다.

6-2) 질의 입력 전에(선행 조건·주의)

  • dataset_id 필수: 업로드·샘플 즉시 업로드 후 녹색 ID가 잡힌 뒤 탭으로 이동한다.
  • 응답 품질과 dataset_type: 한 CSV만 올렸다면 해당 도메인 위주로 질문하는 것이 안전하다. mixed 등 한 데이터셋에 여러 성격이 섞인 경우에만 “SCM·CRM·BI를 한꺼번에” 요약하는 식의 질의가 잘 맞는다. 샘플만 빠르게 돌릴 때는 아래 A(기본) 또는 D~F처럼 범위를 좁힌 질의를 권장한다.
  • 시간: Tier2는 수십 초~최대 3분 걸릴 수 있다. 강의에서는 질의 1개만 실행하고 결과를 읽는 편이 낫다.

Dify를 따로 둔 이유 vs 로컬 Ollama 3.2 (교육생이 “왜 둘 다?”라고 할 때)

Dify (self-hosted) Ollama 3.2 (로컬 LLM)
한 줄 AI 오케스트레이션 — 워크플로 캔버스, HTTP Request로 FastAPI 호출, 변수·조건·LLM 노드 체인, 앱/에이전트 실행 UI 추론 엔진 — 설정된 모델로 채팅/완성 토큰 생성만 담당
이 프로젝트에서 demo/DEMO_SCRIPT의 Studio 시연, 백엔드 /agent/query 가 내부적으로 타는 앱·워크플로 런타임, compact=true로 받은 JSON을 LLM에 넘기는 파이프라인 강의 PC에 깔아 비용·데이터 유출 최소화로 Tier2에 붙인 실제 LLM 백엔드 (클라우드 API 대체 가능)

“(슬라이드) 오늘 데모는 짧은 기간에 올린 1차 축이었고, 납품 완성도보다 계약·파이프라인을 보여 드리는 데 목적이 있었습니다. 마지막으로 앞으로 보강할 방향만 세 줄로 고정하겠습니다.

첫째, 핵심 도메인에서 용어·KPI·데이터 품질을 원천으로 규칙과 스키마를 다시 잡는 것입니다. 어떤 열이 필수인지, dataset_type·period·검사·고객 코드가 무엇을 의미하는지, compact 응답에 무엇을 남길지, Dify 프롬프트와 검증 시나리오까지 같은 도메인 계약으로 묶습니다.

둘째, 그 계약 위에서 RAG로 내부 문서·로그·SOP를 붙이고, 에이전트로 Tier1 API를 도구 호출로 묶는 쪽으로 확장할 준비를 합니다. 관계·메타데이터가 중요해지면 GraphRAG 같은 구조화 지식 쪽도 검토합니다 — 단, 수치는 여전히 Tier1이라는 전제는 유지합니다.

셋째, AI 백엔드와 웹 프로토타입은 그대로 두되, 우선순위는 도메인 규칙 → 스키마·API → 그다음 LLM 확장 순으로 잡는 것이 안전하다고 보시면 됩니다.

(질문 받기) KPI 대시보드나 3축 기능 로드맵은 조직 과제로 남기고, 기술 스택 관점에서는 오늘 말씀드린 도메인 계약 + RAG/에이전트 준비를 기억해 주시면 됩니다.”
