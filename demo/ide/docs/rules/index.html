AI 변경 통제 개발 환경 — 규칙 가이드

IDR Analytics(lis_cursor)에서 Cursor AI를 통제 가능한 공정으로 두기 위한 레이어·문서·게이트 요약입니다. 강의용으로 원본 규칙을 압축·재구성했습니다.

교육용 요약 · 아래 ZIP 패키지로 로컬 Cursor에 복사 적용 가능 · 원문은 저장소 docs/rules/, .cursorrules, .cursor/ 참고

교육생용 규칙 패키지

URL 예: /ide/downloads/idr-cursor-rules-student.zip — 상세 계획은 저장소 docs/plans/student_rules_download_lis_plan.md

아래 링크로 ZIP을 받은 뒤 압축을 풀고, 폴더 루트의 README_교육생.md대로 실습 프로젝트에 복사합니다.

ZIP 다운로드 SHA256 무결성 파일

공인 URL에서 404가 나면 엣지 nginx에서 /ide/가 API와 동일 업스트림인지 확인한 뒤, 운영 표준(docs/plans/lis_public_url_path_map.md §0)대로 로컬 uvicorn·터널·리포 demo/ide를 점검합니다. 서버 컨테이너에 파일을 심는 것이 기본 처방은 아닙니다.

0. 왜 규칙 체계가 필요한가

  • 속도만으로는 부족합니다. AI가 코드를 빨리 써도 아키텍처·타입·성능·보안·운영 기준은 자동으로 맞지 않습니다.
  • 조합이 핵심입니다. 문서(계약) + Gate(상태 머신) + 테스트·정적분석(증적)을 같이 써야 변경 영향과 리뷰 부담이 줄어듭니다.
  • 목표는 “AI 활용”이 아니라 “통제 가능한 공정”입니다. Spring에서 트랜잭션 경계를 명시하듯, 여기서는 Gate와 문서로 작업 경계를 명시합니다.

2. L1 — .cursorrules (진입 계약)

  • 코드 작성 전에 docs/plans/plan.md, docs/CURRENT_WORK_SESSION.md, docs/rules/error_analysis.md 등을 읽도록 강제합니다.
  • 핵심 원칙 예: Dify는 수치 계산 안 함(FastAPI/Pandas), endpoints/*.py에서 from __future__ import annotations 금지(422 이슈), 분석 API에 compact=true, async 일관성, 오류는 error_analysis.md에 기록.
  • 강의 관점: “AI에게 매번 말하지 말고, 저장소에 계약을 박아 둔다”는 패턴입니다.

3. L2·L3 — 커밋 규칙과 스킬(Skill)

L2: git-commit-korean.mdc

  • alwaysApply: true — Cursor가 커밋 메시지를 생성할 때 한국어 Conventional Commits를 쓰도록 유도합니다.
  • .vscode/settings.json의 Copilot 커밋 지시와 같은 취지로 이중화해 둔 경우가 많습니다(도구별로 설정이 갈리므로).

L3: idr-session-workflow/SKILL.md

  • Gate A~E를 표로 압축하고, “Gate B 직후 멈춤” 같은 자주 깨지는 지점을 강조합니다.
  • Rules vs Skills: Rules는 보통 항상/조건부 텍스트 규칙, Skills는 워크플로·도메인 절차를 묶은 참조 모듈에 가깝습니다(실제 Cursor 제품 용어에 맞춰 팀 내에서 통일하면 됩니다).

5. 백엔드 아키텍처·품질 게이트 (L4 + L5)

아키텍처(요지)

  • 2-Tier: 단순·수치 분석은 Tier1(Pandas 등), 자연어·복합 설명은 Tier2(Dify+LLM). ComplexityScorer로 라우팅.
  • async 일관성: 엔드포인트·서비스는 async를 기본으로. CPU-bound 동기 라이브러리는 스레드 풀·ARQ 등으로 빼는 패턴을 문서에 명시.
  • 엔드포인트 파일: TYPE_CHECKING만으로 타입을 숨기면 FastAPI Depends가 런타임에 타입을 못 잡아 422로 이어질 수 있음 → endpoints/*.py에서는 직접 import.

품질 게이트(L5)

  • Ruff — lint + format, line-length = 120 등은 pyproject.toml.
  • mypy strict — pyproject.toml의 [tool.mypy]와 pre-commit 훅의 의존성을 맞춰 “로컬과 훅 결과 불일치”를 줄입니다.
  • pre-commit — 커밋 시 ruff·ruff-format·mypy. Unstaged + stash 롤백 이슈가 실제로 기록되어 있으므로, 커밋 전 변경분 전체 스테이징이 운영 규칙입니다.

6. Dify 연동 규칙 (L4)

  • 역할 분리: CSV 처리·예측·클러스터 등 수치/집계는 FastAPI. Dify는 HTTP Request로 받은 정제된 JSON을 LLM 체인에 넘깁니다.
  • compact=true: 분석 API는 Dify/LLM 컨텍스트 절약을 위해 축약 응답을 지원. 설계 기준(필드 수·대략 크기 등)은 docs/rules/dify_integration.md 참고.
  • 동기 JSON 위주: 예를 들어 202 + job_id 패턴은 Dify 노드에 바로 넣기 어려울 수 있어, 워크플로에는 동기 GET 위주 조합을 권장하는 식으로 문서화되어 있습니다.
강의에서는 “백엔드 계약(JSON)·크기·에러 형태”를 먼저 고정하고 Dify를 얹는 순서를 권합니다.

7. 오류 사례·재발 방지 (error_analysis.md 교육용 발췌)

실제 기록을 카드로만 요약했습니다. 전문은 저장소 문서를 보세요.

구현 직후 세션 전환·테스트 게이트 생략
구현 후 사용자 확인·테스트 계획 없이 CURRENT를 다음 세션으로 통째 교체하면 안 됩니다. → Gate B에서 멈추고, C 이후만 테스트.
“작업 계획”을 Cursor 플랜 도구로만 처리
이 프로젝트에서는 docs/CURRENT_WORK_SESSION.md가 계획의 본 기록입니다. 범용 플랜 파일로 대체하지 않습니다.
pre-commit: unstaged + 훅 자동 수정 롤백
일부만 stage한 채 커밋하면 stash 충돌로 ruff/mypy 수정이 되돌아갈 수 있습니다. → 변경 전부 스테이징, 로컬에서 먼저 format/lint/typecheck.
시크릿·.env* 정책
.env*는 Git에 올리지 않고, 템플릿은 env.example 등 점 없는 파일명으로 관리합니다. 플레이스홀더도 스캐너 오탐을 피할 것.

8. 빠른 참조 체크리스트

세션 시작

  • plan.md 위치 확인 → CURRENT_WORK_SESSION.md Gate 상태 확인
  • docs/rules/error_analysis.md 최근 항목 스캔
  • 구현/리뷰 시 project_context.md, backend_architecture.md, 필요 시 SDD

구현 전(Gate A)

  • CURRENT에 범위·파일·주의사항·완료 기준을 쓴 뒤 사용자 승인

구현 후(Gate B)

  • 구현 요약·변경 파일·제약을 CURRENT에 남기고 멈춤

테스트(Gate C~D)

  • 테스트 범위·명령·우선순위를 문서화 후 승인 → 실행 → 결과 기록

커밋 전(L5)

  • make format && make lint && make typecheck 등 프로젝트 표준 명령 통과
  • 스테이징 누락 없이 커밋(pre-commit stash 롤백 방지)
