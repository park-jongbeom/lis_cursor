AI 변경 통제 개발 환경 — 규칙 가이드

IDR Analytics(lis_cursor) 예제를 따라갈 때, Cursor·VS Code + Claude Code 등 AI 코딩 도구를 통제 가능한 공정으로 쓰기 위한 레이어·문서·게이트 요약입니다. 저장소 원본 규칙을 실습하기 읽기 쉽게 압축했습니다.

사용 중인 도구 탭을 고르세요 · 공통 ZIP 패키지(Cursor용 README 포함) · 세부 원문은 복제한 저장소의 docs/rules/, .cursorrules, .cursor/에서 확인

아래 링크로 ZIP을 받은 뒤 압축을 풀고, 폴더 루트의 README_교육생.md대로 실습 프로젝트에 복사합니다. (Claude Code 사용자는 오른쪽 탭의 경로 매핑을 함께 보세요.)

이 페이지나 ZIP이 열리지 않으면 강사·조교에게 먼저 알려 주세요. 수업 안내(로컬 실습·URL)를 따르면 됩니다. 스스로 점검이 필요할 때만 저장소의 lis_public_url_path_map.md, README_교육생.md를 참고하세요.

0. 왜 규칙 체계가 필요한가

  • 속도만으로는 부족합니다. AI가 코드를 빨리 써도 아키텍처·타입·성능·보안·운영 기준은 자동으로 맞지 않습니다.
  • 조합이 핵심입니다. 문서(계약) + Gate(상태 머신) + 테스트·정적분석(증적)을 같이 써야 변경 영향과 리뷰 부담이 줄어듭니다.
  • 목표는 “AI 활용”이 아니라 “통제 가능한 공정”입니다. Spring에서 트랜잭션 경계를 명시하듯, 여기서는 Gate와 문서로 작업 경계를 명시합니다.

보충: 프롬프트(지시)에서 규칙·금지(통제)로

이 저장소의 규칙 문서들은 처음부터 완성된 설계도가 아니라, 실제로 AI와 함께 일하면서 방향을 바꾼 결과에 가깝습니다.

초기: 프롬프트 엔지니어링 중심

  • 세션마다 “이렇게 해라”는 긴 지시(프롬프트)로 행동을 맞추려 했습니다. 모델·도구·대화 맥락이 바뀔 때마다 문장을 다시 쓰게 됩니다.
  • 잘 되면 빠르지만, 같은 지름길(테스트·승인 생략, 계획을 다른 파일로 분산, 원격 MCP로 배포까지 확대 등)을 반복하기 쉽습니다. 사람이 매번 같은 문장으로 막기엔 한계가 있습니다.

현재: 금지·경계·계약 중심

  • 지금은 “이렇게 하면 안 된다”, “승인 전에는 무엇이 금지인지”, “계획은 어디 한 곳에만”처럼 통제 조건을 저장소 파일(.cursorrules, workflow_gates.md, 스킬, L4 문서)에 박아 둡니다.
  • 그래서 Gate 표에 「승인 전 금지」 열이 있고, L1에는 필독·금지가 길게 잡히며, 틀린 패턴은 error_analysis.md에 쌓아 다음 세션의 입력으로 다시 씁니다.
  • 프롬프트는 여전히 쓰이지만, 재현 가능한 팀 공정의 뼈대는 “잘 써 봐”가 아니라 문서·게이트·훅·사례 기록이 담당합니다.
아래 5개 레이어·하네스·사례 카드는 그 전환을 구체화한 것입니다. “왜 이렇게 복잡해졌나?”는 질문에는 한 번 막지 못했던 실수가 규칙 문장이 됐기 때문이라고 보면 됩니다.

복제한 저장소 기준 경로입니다. 실습할 때 Cursor·에이전트가 이 순서를 전제로 동작합니다.

보충: 에이전트·품질 하네스(운영 장치)란?

테스트 자동화에서 말하는 하네스처럼, AI 에이전트와 사람이 같은 루트·같은 품질 기준을 반복하도록 묶어 둔 장치들입니다. 프롬프트만으로 “매번 잘해 줘”를 반복하기보다, 금지·순서·검증을 고정하는 쪽이 재현성에 낫다는 경험에서 이런 장치를 같이 씁니다. 코딩만 잘하는 것이 아니라 실수가 다시 나오지 않게 아래를 둡니다.

  • 문서 순서 하네스 — 세션 시작 시 plan.md → CURRENT_WORK_SESSION.md → error_analysis.md 를 읽는 순서를 스킬(idr-session-workflow/SKILL.md)로 고정해, 매번 다른 시작을 막습니다.
  • 승인·상태 하네스 — Gate A~E로 “지금은 계획만 / 구현만 / 테스트는 아직”처럼 허용 행동을 나눕니다. 특히 구현 직후(Gate B)에 잠깐 멈추기는 승인 범위를 넘겨 해석하는 실수를 줄이기 위해 스킬에 굵게 넣었습니다.
  • 요약·주입 하네스 — 긴 workflow_gates.md를 에이전트가 매 턴 참고하기 쉬운 한 장(SKILL)으로 압축하고, .cursorrules로 “plans 신설 금지·MCP 범위”처럼 자주 깨지는 줄을 최상단에 다시 적습니다.
  • 도구(MCP) 범위 하네스 — 원격 서버 MCP는 엣지 nginx 설정 확인 중심으로 두고, 컨테이너 재배포·호스트에 파일 심기까지 확대하지 않게 문서·스킬에 표로 고정했습니다.
  • 코드 검증 하네스 — Ruff·mypy·pytest·pre-commit·Makefile(make format 등)으로 “한 번 통과한 기준”을 기계적으로 반복합니다.
  • 회귀·사례 하네스 — AI가 틀렸던 패턴을 docs/rules/error_analysis.md에 날짜·유형으로 쌓아, 다음 세션의 필독 입력으로 재사용합니다. 아래 「규칙을 이렇게 둔 이유」 절이 그 요약입니다.
  • (배포 점검) — 공인 URL 등은 scripts/verify_lis_public_smoke.py, make prod-smoke-ide 같은 스크립트로 동선을 한 번에 검증하는 식의 하네스도 같이 둡니다.

2. L1 — .cursorrules (진입 계약)

  • AI·에디터가 매 세션 참고할 최상위 계약입니다. 코드를 건드리기 전에 어떤 문서를 읽는지, 무엇을 하지 말아야 하는지가 한곳에 모여 있습니다.
  • 구체 목록(필독 경로, 금지 패턴 등)은 복제한 저장소의 .cursorrules 원문을 열어 확인하세요. 이 페이지는 개념·구조 안내용이라 모든 세부 규칙을 적지 않습니다.
  • 워크플로는 workflow_gates.md가 단일 원본이며, 이 저장소에서 Cursor로 함께 작업하는 사람 모두에게 동일하게 적용됩니다.
  • 실습에서 기억할 점: AI에게 매번 말로 되풀이하지 말고, 저장소에 적어 둔 계약을 기준으로 맞추는 방식입니다.

3. L2·L3 — 커밋 규칙과 스킬(Skill)

L2: git-commit-korean.mdc

  • alwaysApply: true — Cursor가 커밋 메시지를 생성할 때 한국어 Conventional Commits를 쓰도록 유도합니다.
  • .vscode/settings.json의 Copilot 커밋 지시와 같은 취지로 이중화해 둔 경우가 많습니다(도구별로 설정이 갈리므로).

L3: idr-session-workflow/SKILL.md

  • Gate A~E를 표로 압축하고, “Gate B 직후 멈춤” 같은 자주 깨지는 지점을 강조합니다.
  • Rules vs Skills: Rules는 보통 항상/조건부 텍스트 규칙, Skills는 워크플로·도메인 절차를 묶은 참조 모듈에 가깝습니다(실제 Cursor 제품 용어에 맞춰 팀 내에서 통일하면 됩니다).

5. 자동 검증·품질 습관 (L5)

AI가 만든 코드도 사람과 같은 기준으로 검사한다는 뜻을 이해하면 됩니다. 아직 API를 짜지 않는 실습 단계에서도, 린트·타입·테스트가 왜 붙는지 감을 잡는 데 도움이 됩니다.

  • Ruff — 포맷·린트. 설정은 pyproject.toml 등에 둡니다.
  • mypy — 정적 타입 검사. 로컬과 pre-commit 훅 결과가 어긋나지 않게 의존성을 맞춥니다.
  • pytest — 실행 기반 검증(테스트가 있는 경우).
  • pre-commit — 커밋 직전에 위 도구를 돌려 실패를 미리 잡습니다. 일부 파일만 올린 채 커밋하면 수정이 되돌아가는 일이 있어, 바꾼 파일은 모두 스테이징하는 습관이 문서에 적혀 있습니다.
백엔드 아키텍처(예: 2-Tier, 엔드포인트 패턴)는 이 요약 페이지에 없습니다. 해당 과제를 할 때 docs/rules/backend_architecture.md를 열어 보세요.

6. (심화·참고) 연동·오케스트레이션 문서

지금 단계에서는 읽지 않아도 됩니다. 나중에 API·외부 도구·워크플로를 연결하는 과제를 할 때 아래 원문을 엽니다.

  • 연동·요약 파이프라인 등: docs/rules/dify_integration.md
  • 백엔드 레이어·async·엔드포인트 관례: docs/rules/backend_architecture.md
0~5절과 체크리스트만 먼저 익혀도 됩니다. 6절은 연동·백엔드 과제를 시작할 때 열어 보세요.

보충: 규칙을 이렇게 둔 이유 (실제로 겪은 사례)

아래는 이 프로젝트를 만들며 실제로 발생한 일을 바탕으로, 왜 문서·게이트·도구 설정이 이렇게 굳었는지 짧게 정리한 것입니다. 전문·날짜별 기록은 저장소 docs/rules/error_analysis.md에서 확인하세요.

규칙 문서는 “기획서 한 장”이 아니라 누적입니다
처음에는 긴 프롬프트로 행동을 맞추려 했지만, 같은 오판이 반복될 때마다 그때그때의 금지·절차를 error_analysis.md에 적고, .cursorrules·workflow_gates.md·SKILL.md·L4 문장으로 다시 흡수했습니다. 그래서 지금 보는 규칙은 “원칙 선언”이라기보다 통제가 필요했던 지점의 기록에 가깝습니다.
Gate B 직후에 꼭 멈춘다 (2026-03-27)
「진행해라」는 구현(Gate A→B)까지만 동의로 해석해야 하는데, 에이전트가 곧바로 make test·이력 갱신까지 한 턴에 진행한 사례가 있었습니다. 그래서 SKILL.md에 Gate B 직후 필수 멈춤을 블록으로 넣고, .cursorrules에도 같은 취지를 반복했습니다.
작업 계획은 CURRENT 한 곳 (2026-03-26)
“계획을 써 달라”는 요청에 Cursor 내장 플랜 도구로 별도 파일만 만들고, 이 프로젝트의 본 기록인 CURRENT_WORK_SESSION.md를 비운 적이 있었습니다. 그래서 “상세 계획은 CURRENT만”이 규칙으로 고정되었습니다.
docs/plans/에 세션 계획 파일을 새로 만들지 않는다 (2026-03-28)
Gate A 상세를 docs/plans/<새 파일>.md에만 쓰면 단일 진실 원천이 깨집니다. 비슷한 실수가 나와 상세는 항상 CURRENT, plans/는 WBS·경로 정본·강의 참조용으로만 쓰도록 나눴습니다.
커밋 전 변경분 전부 스테이징 + 로컬에서 먼저 포맷/린트 (2026-03-26~27)
일부만 stage한 채 커밋하면 pre-commit이 stash를 쓰다가 자동 수정이 롤백되어 같은 오류가 반복되었습니다. 그래서 “make format 등 선행 → git add로 전부”가 운영 습관으로 문서화되었습니다.
MCP(원격)는 엣지 nginx 위주, 배포로 확대 해석하지 않는다 (2026-03-27~28)
«프록시 확인» 지시가 원격 파일 업로드·compose 변경까지 확대된 사례가 있었습니다. ga-nginx 안 설정 읽기·(명시 시) 그 conf 수정만 허용하고, 앱 컨테이너 조작은 금지하는 표가 스킬·컨텍스트 문서에 들어갔습니다.
공인 FastAPI 경로는 «로컬 우회»가 팀 표준 (2026-03-28)
문서에 패턴 A/B가 나란히 있으면 AI가 서버에서 이미지 재빌드를 먼저 제안하는 일이 반복되었습니다. lis_public_url_path_map.md 맨 앞 절에 운영 표준을 한 줄로 고정했습니다.
endpoints/*.py에서 from __future__ import annotations 금지 (설계·재발 방지)
FastAPI Depends가 런타임에 타입을 못 잡아 422로 이어진 패턴을 막기 위해 아키텍처 규칙에 명시했습니다. 자세한 이유는 docs/rules/backend_architecture.md를 보세요.
비밀번호 검증은 passlib 래핑 대신 bcrypt 직접 (2026-03-26)
라이브러리 조합에서 런타임 오류가 나와, 검증·해시 경로를 정리하고 의존성을 명시하는 쪽으로 바꿨습니다. 세부는 error_analysis.md 해당 날짜 항목을 참고하세요.

7. 오류 사례·재발 방지 (error_analysis.md 요약)

팀에서 실제로 적어 둔 사례를 카드로만 옮겼습니다. 자세한 내용은 복제한 저장소의 같은 파일을 열어 보세요.

구현 직후 세션 전환·테스트 게이트 생략
구현 후 담당자 확인·테스트 계획 없이 CURRENT를 다음 세션으로 통째 바꾸면 안 됩니다. → Gate B에서 멈추고, C 이후만 테스트합니다.
Gate A 승인만으로 구현 후 곧바로 테스트(C/D)까지 진행
「진행해라」가 Gate A(구현)까지만 허용한 것일 수 있습니다. 구현이 끝난 뒤 테스트 단계(C) 승인 없이 make test 등을 돌리면 안 됩니다.
“작업 계획”을 Cursor 플랜 도구로만 처리
이 프로젝트에서는 docs/CURRENT_WORK_SESSION.md가 계획의 본 기록입니다. 범용 플랜 파일로 대체하지 않습니다.
세션 상세 계획을 docs/plans/<신규>.md에만 작성
Gate A·C 상세는 CURRENT에만 둡니다. docs/plans/는 전체 일정·경로 안내용입니다.
MCP·공인 경로: idr-fastapi 재빌드·호스트 배포를 기본 처방으로 확대
ga-server MCP는 엣지 nginx(conf) 범위만. 공인 FastAPI·/ide는 로컬 uvicorn·터널·리포 demo/ide가 운영 표준입니다(lis_public_url_path_map.md 정본).
공인 /ide/… 를 «데모와 다른 제품»으로 오해
동일 호스트·경로 분리입니다. /·/api/v1/·/ide/는 같은 FastAPI 업스트림을 가리켜야 합니다.
pre-commit: unstaged + 훅 자동 수정 롤백
일부만 stage한 채 커밋하면 stash 충돌로 ruff/mypy 수정이 되돌아갈 수 있습니다. → 변경 전부 스테이징, 로컬에서 먼저 format/lint/typecheck.
시크릿·.env* 정책
.env*는 Git에 올리지 않고, 템플릿은 env.example 등 점 없는 파일명으로 관리합니다. 플레이스홀더도 스캐너 오탐을 피할 것.

8. 빠른 참조 체크리스트

세션 시작

  • plan.md 위치 확인 → CURRENT_WORK_SESSION.md Gate 상태 확인
  • docs/rules/error_analysis.md 최근 항목 스캔
  • 구현/리뷰 시 project_context.md, backend_architecture.md, 필요 시 SDD

구현 전(Gate A)

  • CURRENT에 범위·파일·주의사항·완료 기준을 쓴 뒤, 과제·수업 절차에 따라 담당자(강사·조교 등) 승인을 받음

구현 후(Gate B)

  • 구현 요약·변경 파일·제약을 CURRENT에 남기고 멈춤

테스트(Gate C~D)

  • 테스트 범위·명령·우선순위를 문서화 후 승인 → 실행 → 결과 기록

커밋 전(L5)

  • make format && make lint && make typecheck 등 프로젝트 표준 명령 통과
  • 스테이징 누락 없이 커밋(pre-commit stash 롤백 방지)

VS Code + Claude Code — 같은 규칙, 다른 경로

규칙 내용(Gate A~E, L4·L5, 오류 기록)은 Cursor 탭과 같습니다. 이 탭에서는 Claude Code·VS Code에서 쓰는 파일 위치만 다르게 안내합니다. 「하네스」·「규칙 배경·사례」 보충 절은 Cursor 탭에만 있으니, 개념은 거기서 먼저 읽고 이 탭으로 돌아와도 됩니다.

도구 버전에 따라 경로가 바뀌면 해당 제품 공식 문서를 보고, 수업에서 받은 템플릿이 있으면 그것을 따르세요.
